# 第36篇 T36-同花顺Excel导入债券估值

## 研究背景

债券估值数据是固定收益研究的核心基础设施。无论是日常的久期监控、信用利差分析，还是定价模型构建，都离不开收益率、久期、评级等基础估值指标。然而，传统上这些数据的获取方式是研究员手动在金融数据终端中逐个查询——这种方式不仅耗时费力，而且随着固收市场债券数量的激增，已经无法满足日常数据更新的需求。

2025年行业趋势显示，RPA技术和AI正在深度赋能固收投研工作流。根据公开报道，部分机构已将文档处理效率从3小时压缩至3分钟，分析师得以将精力投入到核心投研工作。大成基金等机构也在积极建设"固收全链路数智一体化平台"，这一趋势清晰地指向一个方向：数据采集自动化是固收研究现代化的必经之路。

本项目正是在这一背景下诞生的实践案例。项目的核心诉求非常明确：如何通过Python自动化脚本，批量从同花顺iFinD终端获取债券估值数据？这看似是一个简单的技术问题，但深入实施后会发现，其中蕴含着工具选择、容错设计、数据处理等多个工程化思维的应用。

## 核心方法与框架

### 三模块流水线架构

面对"批量获取债券估值数据"这一任务，最直观的思路可能是写一个"大而全"的脚本，把所有操作都塞进去。但本项目采用了一种更清晰的设计——将复杂的任务拆解为三个独立模块，形成流水线架构：

```
模块一：模板生成 → 模块二：公式填充 → 模块三：计算保存
```

这种设计的好处在于：

**职责分离**：每个模块只负责一件事情。模块一关心"查什么债券"，模块二关心"用什么公式"，模块三关心"如何计算"。这种分离使得每个模块都可以独立测试、独立优化，降低了系统的复杂度。

**断点续传**：如果模块三执行失败（比如Excel崩溃），不需要从模块一重新开始，只需要重新执行模块三即可。这对于处理大规模数据时尤为重要——几小时的计算不至于因为最后一步出错而前功尽弃。

**灵活扩展**：如果未来需要更换数据源（比如从iFinD换成Wind），只需要修改模块二的公式部分，其他模块可以完全复用。

### 工具分层策略：用对的工具做对的事

这是本项目最具参考价值的设计思想之一。在Excel自动化处理领域，Python有多个可选的工具库，但每个工具都有其能力边界：

| 工具 | 优势 | 劣势 |
|------|------|------|
| openpyxl | 轻量级、无需安装Office、读写速度快 | 无法执行公式计算、无法调用Excel插件 |
| win32com | 完整的Excel功能、支持公式计算和插件 | 依赖Windows和Office、效率较低、需要Excel进程 |

本项目没有执着于"用一个工具搞定一切"，而是根据任务特性选择最合适的工具：

- **模块一（模板生成）**和**模块二（公式填充）**使用openpyxl：这两个阶段只需要读写Excel单元格、填充字符串，不需要任何Excel的高级功能。openpyxl的轻量级特性在这里发挥得淋漓尽致——快速、稳定、不依赖Office环境。

- **模块三（计算保存）**使用win32com：这一阶段需要Excel真正执行公式计算，还需要调用iFinD插件从服务器获取数据。只有win32com的COM接口才能实现这些功能。

这种"工具分层"的思想可以广泛应用于其他技术选型场景。核心原则是：理解每个工具的能力边界，让它们各司其职、组合发挥最大效用。比如数据处理可以用pandas做清洗、用专门的数据库做存储、用可视化工具做展示；又比如网络请求可以用requests做简单调用、用aiohttp做高并发场景。

### SQL存续期筛选：在数据源头做好过滤

固收市场存在大量已到期或退市的债券，如果对这些债券也进行估值数据查询，不仅浪费API调用额度，还会产生无效数据。本项目在模块一的SQL查询阶段就设计了存续期筛选逻辑：

```sql
SELECT DISTINCT dt, trade_code
FROM temp_dt, basicinfo_credit
WHERE dt.value_date < dt.query_date
  AND dt.maturity_date > dt.query_date
  AND dt.list_end_date IS NULL OR dt.list_end_date > dt.query_date
  AND dt.issue_fail_flag = 'N'
```

这个筛选逻辑体现了"早过滤"原则：越早过滤无效数据，后续环节的效率越高。很多性能问题可以通过在数据源头做好筛选来解决，而不是在下游处理时才发现数据无效。

同时，项目还考虑了信用债、金融债、ABS三类债券的数据源差异，分别从不同的基础信息表中查询，确保数据的完整性。

## 技术亮点详解

### 亮点一：重试机制与异常处理

COM接口（win32com）本质上是不稳定的。Excel进程可能因为资源冲突而崩溃，COM调用可能因为莫名其妙的系统原因而失败。如果遇到一次失败就放弃，批量处理将变得极其脆弱。

本项目设计了一个重试装饰器来解决这个问题：

```python
def retry_on_exception(max_retries=5, delay=30):
    def decorator(func):
        def wrapper(*args, **kwargs):
            for attempt in range(max_retries):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if attempt < max_retries - 1:
                        time.sleep(delay)
                        continue
                    raise e
        return wrapper
    return decorator
```

最多5次重试，每次失败后等待30秒。这种"重复冗余设计"的思路非常实用：与其追求单次执行的完美，不如设计合理的容错机制。在实际运行中，这个机制显著提升了批量处理的稳定性——偶发的失败不会导致整个任务中断。

### 亮点二：60秒等待机制

iFinD Excel插件需要在Excel启动后从服务器加载数据字典和配置信息，这个过程需要时间。如果在插件未完全加载时就执行公式计算，公式会返回错误值（如"#N/A"或"FFFF"）。

本项目的解决方案很简单但很有效：

```python
excel.Visible = False
excel.DisplayAlerts = False
workbook = excel.Workbooks.Open(file_path)
time.sleep(60)  # 等待iFinD插件加载完成
```

60秒的等待时间可能看起来有些保守，但考虑到批量处理场景下稳定性比速度更重要，这个等待是值得的。实际使用中，可以根据网络状况和终端响应速度适当调整这个参数。

### 亮点三：大数据分片处理策略

Excel有行数限制（1,048,576行），而且单个文件过大时打开和保存都会变慢。本项目设计了每30万行分割一个Excel文件的策略：

```python
rows_per_file = 300000
for i, chunk in enumerate(range(0, total_rows, rows_per_file)):
    df_chunk = df.iloc[chunk:chunk + rows_per_file]
    # 保存到独立的Excel文件
```

这个策略有两个好处：

1. **规避Excel限制**：无论有多少数据，都能被分割成多个合规的Excel文件
2. **便于并行处理和断点续传**：每个文件可以独立处理，某个文件失败不影响其他文件

### 亮点四：iFinD公式参数的差异化设计

债券市场分为银行间市场和交易所市场，两个市场的估值数据获取方式有所不同。本项目在公式填充时体现了这种差异：

- **银行间债券**：使用`ths_evaluate_yield_yy_bond`指标直接查询
- **交易所债券**：使用`ths_evaluate_yield_cb_bond`指标，并传入参数102

```python
# 银行间债券收益率公式
formula_yy = f'=@thsiFinD("ths_evaluate_yield_yy_bond",{trade_code},{date})'

# 交易所债券收益率公式（需要参数102）
formula_cb = f'=@thsiFinD("ths_evaluate_yield_cb_bond",{trade_code},{date},102)'
```

这种差异化设计需要对两个市场的规则有清晰的理解。如果用错参数，公式可能返回错误值或不完整的数据。

## 实际应用场景

### 场景一：日常估值数据更新

固收研究团队通常需要定期（每周或每月）更新所有存续债券的历史估值数据。使用本项目的三模块流水线：

1. 准备日期列表（temp_dt表）和债券基础信息表
2. 运行模块一生成Excel模板
3. 运行模块二填充iFinD公式
4. 运行模块三执行计算并保存

整个过程可以无人值守运行，节省大量手工操作时间。

### 场景二：债券定价分析

在进行债券定价分析时，需要获取大量债券的收益率、久期数据进行比较和建模。本项目的批量获取能力可以支持这类分析需求，且数据以静态值形式保存，便于后续的定量分析。

### 场景三：信用利差研究

信用利差研究需要获取不同评级债券的收益率数据，计算利差并分析其变化趋势。本项目可以一次性获取所有目标债券的收益率数据，为利差分析提供基础数据支持。

### 场景四：其他金融数据批量获取

虽然本项目针对债券估值数据设计，但技术框架可以推广到其他需要通过Excel插件获取的金融数据场景，如：

- 股票财务指标批量获取
- 基金净值历史数据
- 宏观经济指标时序数据

核心的"三模块流水线+工具分层"架构可以直接复用，只需要修改SQL查询逻辑和iFinD公式即可。

## 心得要点

### 心得一：工具分层——根据任务特性选择最合适的工具

在项目初期，我曾纠结于应该用openpyxl还是win32com。openpyxl轻量但功能有限，win32com强大但依赖重。最终的答案是：**不要纠结于选一个"最好的"工具，而是根据任务特性选"最合适的"工具，然后组合使用。**

这个思路可以迁移到很多技术选型场景：

- 数据存储：小数据用SQLite，大数据用PostgreSQL，时序数据用InfluxDB
- Web框架：简单接口用Flask，完整应用用Django，高性能用FastAPI
- 任务调度：单机用Celery，分布式用Airflow

核心是理解每个工具的**能力边界**，而不是追求一个"万能"的解决方案。

### 心得二：容错设计——为不可靠的外部依赖建立防护机制

本项目依赖两个外部组件：iFinD终端和Excel COM接口。这两者都是"不可控"的外部依赖——iFinD可能因为网络问题响应缓慢，COM接口可能因为进程冲突而失败。

与其期望外部依赖永远稳定，不如设计合理的容错机制：

1. **重试机制**：承认失败可能发生，但通过重试来提高成功率
2. **等待机制**：承认某些操作需要时间，给系统足够的准备时间
3. **分片处理**：承认长时间运行可能出问题，通过分片来降低单次失败的影响范围

这种"防御性编程"的思维适用于所有涉及外部API调用、网络请求、第三方服务的场景。关键原则是：**识别系统中的脆弱环节，为这些环节设计冗余保护。**

### 心得三：数据预处理——在数据源头做好筛选和清洗

本项目在SQL查询阶段就进行了存续期筛选，而不是在获取数据后再过滤。这看似是一个小优化，但体现了"源头治理"的重要思想：

- **效率**：越早过滤，后续环节处理的数据量越小，效率越高
- **成本**：API调用是有成本的（无论是金钱成本还是时间成本），减少无效调用就是降低成本
- **质量**：减少无效数据进入系统，也就减少了后续出错的可能性

这个原则在数据工程中尤为重要。ETL流程的设计应该遵循"早过滤"原则，让每一步都只处理"有价值"的数据。

### 心得四：理解业务规则的差异化

本项目中最容易被忽视但又最重要的细节是银行间和交易所市场的公式参数差异。如果不理解这个业务规则，代码可能"看起来正确"但实际产出错误数据。

这提醒我们：技术自动化必须建立在对业务规则的深刻理解之上。在开发任何数据采集系统之前，先搞清楚数据源的规则和差异，比埋头写代码更重要。

## 局限性与改进方向

### 当前局限

1. **Windows依赖**：win32com只能在Windows上运行，且需要安装Microsoft Office Excel，限制了跨平台部署的可能性。

2. **iFinD终端依赖**：必须安装并登录同花顺iFinD终端，需要相应的账号权限。如果终端未登录或网络中断，数据获取将失败。

3. **串行处理**：当前设计是串行处理多个Excel文件，在大数据量时效率有限。理论上可以并行处理多个文件，但需要注意Excel进程的资源冲突问题。

4. **参数硬编码**：等待时间（60秒）、重试次数（5次）、分片大小（30万行）都是硬编码在代码中。不同环境可能需要不同的参数配置。

5. **缺乏进度反馈**：批量处理时缺乏进度显示和日志记录，难以追踪处理状态和定位问题。

6. **回传数据库功能未实现**：当前数据保存在Excel文件中，还需要手动导入数据库。如果能自动回传数据库，整个流程会更加自动化。

### 改进建议

1. **配置文件化**：将硬编码参数提取到配置文件中，便于不同环境下的调整。

2. **日志增强**：增加详细的日志记录，包括处理进度、成功/失败统计、错误详情等。

3. **并行处理**：探索使用多进程或多服务器并行处理的可能性，提升大数据量场景下的效率。

4. **数据库回传模块**：开发模块四，实现计算结果自动回传数据库的功能，形成完整的数据闭环。

## 延伸思考

### 从数据获取到数据治理

本项目的本质是解决"数据获取"问题，但在更大的视角下，这只是数据治理链条中的一个环节。一个完整的固收数据平台应该包括：

- 数据采集：从各种数据源（Wind、iFinD、中债、交易所）获取原始数据
- 数据清洗：处理异常值、填补缺失值、统一数据格式
- 数据存储：设计合理的数据模型，支持高效查询
- 数据服务：提供API接口，供投研系统调用

本项目解决了数据采集的自动化问题，但后续的清洗、存储、服务环节还需要进一步的工程化建设。

### 自动化与人工的边界

虽然自动化能显著提升效率，但也需要警惕过度自动化的风险。数据质量异常、数据源变更、业务规则调整等情况都需要人工介入。理想的状态是：

- 常规操作自动化，释放人力
- 异常情况人工处理，保证质量
- 定期检查自动化流程的运行状态

### 技术债的平衡

本项目中存在一些"能用但不完美"的设计，比如硬编码的参数、简陋的错误处理。在追求快速交付的同时，需要权衡技术债的积累。关键原则是：核心路径优先保证质量，边缘功能可以适度妥协。
