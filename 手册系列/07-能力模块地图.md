# 第 7 册：能力模块地图

## 1. 为什么还要单独写能力地图

如果前六册讲的是“信用研究怎么想”，那这一册讲的就是“信用研究可以拆成哪些可复用能力”。这一步很重要，因为后续真正进入项目级手册时，不应该从零写，而应该从能力模块组合开始。

## 2. 能力模块的基本定义

能力模块不是具体项目，而是可迁移的工作能力单元。它应该满足三个条件：

1. 能独立描述输入和输出。
2. 能跨主体、跨场景复用。
3. 能被组合到更大的工作流中。

## 3. 常见模块类型

- 数据采集模块：解决信息从哪里来。
- 数据标准化模块：解决口径统一和结构整理。
- 文本分析模块：解决公告、年报、条款和政策文本的抽取。
- 指标计算模块：解决可比较的定量表达。
- 监控预警模块：解决持续跟踪和风险提示。
- 交付组织模块：解决结果如何变成可阅读的研究材料。

## 4. 为什么模块化重要

模块化的意义在于减少重复建设。一个成熟的研究体系，不应该每做一个新主题就重写一套流程，而是应该复用已有模块，再按需要组合。

## 5. 模块如何组合成工作流

一个典型工作流通常是：采集 -> 清洗 -> 标准化 -> 分析 -> 预警 -> 交付。

但不同主题会在其中插入不同模块，比如文本密集的主题会加强文本分析，监控型主题会加强时间序列和预警，评级型主题会加强指标比较和人工复核。

## 6. 这册的作用

这一册不是项目实现手册，而是后续项目级手册的“上位目录”。它的任务是告诉技术人员：以后看到具体项目，不要从头理解，而要先判断它调用了哪些能力模块。
