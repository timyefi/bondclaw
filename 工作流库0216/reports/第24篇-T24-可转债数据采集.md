# 第24篇 可转债数据采集系统：构建稳定高效的金融数据管道

## 一、项目背景与业务需求

### 1.1 可转债市场的投资价值

可转换公司债券（可转债）是中国债券市场的重要品种，兼具债性和股性，是资产配置中独特的投资标的。截至2024年底，沪深两市存续可转债约550只，市场规模超过8000亿元。

可转债的投资价值体现在：

**下行保护**：作为债券，有还本付息的保障，下跌空间有限
**上行弹性**：可以转换为股票，享受股价上涨收益
**条款博弈**：下修条款、强赎条款、回售条款等提供额外的收益来源
**流动性优势**：相比纯债，可转债交易更为活跃

### 1.2 数据采集的业务痛点

可转债投研需要持续跟踪大量数据，但数据采集面临诸多挑战：

**数据源分散**：可转债数据分布在多个平台（同花顺、Wind、东方财富等），格式不统一，需要人工整合。

**更新频率高**：可转债价格、溢价率、转股价值等指标每日变动，需要每日更新才能支持投资决策。

**历史数据缺失**：部分平台只提供近期数据，历史数据回溯困难，影响策略回测。

**指标计算复杂**：转股溢价率、纯债溢价率等指标需要多个基础数据协同计算，手工容易出错。

### 1.3 项目目标

本项目旨在构建一套完整的可转债数据采集系统，实现以下目标：

- **全量覆盖**：覆盖所有存续可转债，包括已退市的可转债历史数据
- **高频更新**：支持每日自动采集，保持数据时效性
- **增量更新**：只采集缺失日期的数据，避免重复请求
- **质量保障**：完善的数据校验和备份机制

## 二、系统架构与数据源

### 2.1 整体架构

系统采用"类封装+批量采集"的架构设计：

```
同花顺iFinD API -> THSBondDataCollector类 -> MySQL数据库 -> 本地备份
```

**核心模块**：
- `src/bond_data_collector.py`：THSBondDataCollector类，封装所有采集逻辑
- `config.py`：配置文件，管理账号、数据库连接、采集参数

**数据流**：
1. 同花顺登录验证
2. 获取可转债代码列表
3. 检查数据库已有数据
4. 计算缺失日期范围
5. 批量采集缺失数据
6. 写入数据库
7. 生成采集日志

### 2.2 数据源选择：同花顺iFinD

选择同花顺iFinD作为数据源的原因：

**数据质量**：
- 可转债指标计算准确，与官方披露一致
- 历史数据完整，支持回溯多年
- 数据更新及时，T日数据T日可获取

**接口稳定**：
- Python SDK成熟，文档完善
- 接口调用稳定，故障率低
- 技术支持响应及时

**成本可控**：
- 相比Wind等竞品，同花顺性价比更高
- 支持双账号配置，应对限流

### 3.3 增量更新检测机制

增量更新是系统效率的核心，设计如下：

**第一步：获取可转债生命周期**
```python
def get_bond_lifecycle(self, code):
    # 从数据库或接口获取上市日期、摘牌日期
    list_date = self.get_list_date(code)
    delist_date = self.get_delist_date(code)
    return list_date, delist_date
```

**第二步：检查已有数据**
```python
def check_existing_data(self, code):
    query = f"""
        SELECT DISTINCT dt
        FROM marketinfo_equity1
        WHERE code = '{code}'
    """
    existing_dates = pd.read_sql(query, self.engine)['dt'].tolist()
    return set(existing_dates)
```

**第三步：计算缺失日期**
```python
def get_missing_dates(self, code, list_date, delist_date):
    # 获取交易日历
    trading_days = self.get_trading_days(list_date, delist_date)

    # 获取已有日期
    existing_dates = self.check_existing_data(code)

    # 计算缺失日期
    missing_dates = [d for d in trading_days if d not in existing_dates]

    return missing_dates
```

**效率提升**：
- 首次全量采集：约2小时（550只可转债 x 3年历史）
- 日常增量更新：约5分钟（550只可转债 x 1天新增）

### 3.5 批量数据处理

大规模数据采集需要精细的批处理设计：

**批次参数**：
```python
BATCH_SIZE = 100      # 每批处理100个可转债代码
REQUEST_DELAY = 0.2   # 请求间隔0.2秒
MAX_RETRY = 3         # 最大重试3次
```

**批处理逻辑**：
```python
def process_batch(self, codes, dates):
    results = []
    for i, code in enumerate(codes):
        try:
            data = self.collect_bond_data(code, dates)
            results.append(data)

            # 请求间隔
            time.sleep(REQUEST_DELAY)

            # 日志记录
            if (i + 1) % 10 == 0:
                self.logger.info(f"已处理 {i+1}/{len(codes)} 只可转债")

        except Exception as e:
            self.logger.error(f"采集 {code} 失败: {e}")
            # 重试逻辑
            for retry in range(MAX_RETRY):
                try:
                    data = self.collect_bond_data(code, dates)
                    results.append(data)
                    break
                except:
                    time.sleep(1)

    return results
```

### 3.6 幂等性设计：ON DUPLICATE KEY UPDATE

数据库写入采用幂等设计，确保重复执行不会产生问题：

```python
def save_to_database(self, df):
    insert_sql = f"""

        VALUES (%s, %s, %s, %s, %s, %s, %s, %s, %s, %s, %s, %s)
        ON DUPLICATE KEY UPDATE
            close = VALUES(close),
            amount = VALUES(amount),
            pure_bond_premium_rate = VALUES(pure_bond_premium_rate),
            conversion_premium_rate = VALUES(conversion_premium_rate),
            pure_bond_ytm = VALUES(pure_bond_ytm),
            conver_pe = VALUES(conver_pe),
            stock_pe = VALUES(stock_pe),
            conver_pb = VALUES(conver_pb),
            stock_pb = VALUES(stock_pb),
            un_conversion_balance = VALUES(un_conversion_balance),
            updated_at = NOW()
    """
```

**设计价值**：
- 重复执行不会产生重复记录
- 支持数据修正（重新采集覆盖旧数据）
- 采集任务可以安全地中断和恢复

## 七、心得要点

### 7.1 增量更新的高效设计

项目实现了完整的增量更新机制，这是金融数据采集系统的核心竞争力：

**设计思想**：
1. 先查询数据库已有数据
2. 对比交易日历找出缺失日期
3. 只采集缺失部分

**效率对比**：
- 全量采集：每日550只 x N年历史，耗时数小时
- 增量更新：每日550只 x 1天新增，耗时数分钟

**代码实现**：
```python
dates_to_fetch = [
    day for day in trading_days
    if code_start_date <= day <= code_end_date
    and day not in existing_dates
]
```

这种设计使得系统可以每日运行，保持数据时效性，同时避免不必要的API调用。

### 7.2 ON DUPLICATE KEY UPDATE的幂等设计

数据库写入使用INSERT ... ON DUPLICATE KEY UPDATE语句，实现了幂等性：

**幂等性的价值**：
- 采集任务可以安全地重复执行
- 数据修正时直接覆盖旧值
- 支持部分重采（只重新采集特定日期）

**代码实现**：
```sql
INSERT INTO table (...) VALUES (...)
ON DUPLICATE KEY UPDATE
    close = VALUES(close),
    amount = VALUES(amount),
    ...
```

这一设计使得数据采集从"一次性任务"变成"可重复执行的任务"，大大提高了系统的可靠性。

- 增加Wind数据接口作为备份

- 单一数据源故障时自动切换

**数据质量校验模块**：

**断点续传功能**：
- 记录采集进度

- 大幅提升大规模采集的可靠性

**配置外置**：
- 将数据库配置、账号配置移至环境变量

2. **可靠性保障**：双账号轮换、幂等写入、完善日志确保系统稳定运行

4. **可扩展性**：模块化设计便于扩展到其他金融数据采集场景

对于投研团队而言，这套系统将可转债数据获取从"手工Excel操作"提升到"自动化数据库服务"，使研究员能够专注于数据分析和策略研究，而非数据整理。这正是数据工程赋能金融研究的价值所在。

**复用价值**：
本系统的核心算法（增量更新检测、ON DUPLICATE KEY UPDATE、批量采集框架）可直接复用于：
- 股票行情数据采集
- 基金净值数据采集
- 其他金融时间序列数据采集
