# AI辅助固收研究系列报告 第2篇：T02-每日脚本任务

> **研究领域**: 固定收益研究基础设施
> **研究目的**: 构建债券研究的自动化数据基础设施，将研究员从繁琐的数据收集工作中解放出来
> **适用场景**: 日常数据跟踪/舆情监控/指标计算
> **更新频率**: 每日多时段自动执行

---

## 1. 研究背景与问题

### 1.1 为什么需要这项研究？

固定收益研究是一项高度依赖数据的工作。无论是信用分析、利率研究还是投资组合管理，都需要持续跟踪各类市场数据、发行人舆情、宏观经济指标等信息。传统的研究方式中，研究员每天花费大量时间在数据收集和整理上——打开Wind终端导出数据、从各网站复制表格、手动更新Excel模板……这些重复性的"脏活累活"不仅效率低下，而且容易出错。

更关键的是，数据的时效性直接影响研究质量。当某发行人出现负面舆情时，如果研究员还在依赖每日一次的手动数据更新，很可能错过最佳应对窗口。在信用风险日益复杂的当下，构建一套能够实时捕获、自动处理、及时预警的数据基础设施，已成为固定收益研究的迫切需求。

### 1.2 核心研究问题

本项目的核心研究问题是：**如何构建一套稳定、高效、可维护的自动化数据采集与处理系统，支持固定收益研究的日常需求？**

具体分解为以下几个子问题：
1. **多源数据整合**：如何统一对接THS API、RatingDog舆情API、交易所数据、土地交易数据等异构数据源？
2. **舆情智能处理**：如何对海量的舆情内容进行自动化筛选、清洗和摘要，提取有价值的信息？
3. **数据更新策略**：如何设计增量更新机制，确保数据完整性的同时避免重复？
4. **任务调度设计**：如何根据数据源特性和业务需求，合理安排任务的执行时间？

### 1.3 研究价值

本项目的价值体现在三个层面：

**效率层面**：自动化数据采集和处理，节省研究员每天数小时的数据整理时间。以舆情监控为例，系统每日自动获取数百条舆情数据，经过AI摘要处理后直接入库，研究员无需手动浏览和整理。

**质量层面**：标准化数据处理流程，减少手动操作带来的错误。增量更新机制确保数据一致性，模糊匹配算法提高发行人识别准确率。

**风险层面**：实时舆情监控帮助及时发现和预警潜在风险事件。滚动窗口统计可追踪发行人近180天的负面舆情累计情况，为信用风险评估提供量化依据。

## 2. 研究思路与方法

### 2.1 整体研究框架

本项目的整体架构采用经典的ETL（Extract-Transform-Load）模式，将系统分为数据采集层、数据处理层和数据存储层三个层次。

```
┌─────────────────────────────────────────────────────────────┐
│                     数据源层（Extract）                       │
│  THS API │ RatingDog API │ 交易所API │ 土地数据API │ ...    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     数据处理层（Transform）                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐    │
│  │ 数据清洗  │→│ 格式转换  │→│ AI处理   │→│ 指标计算  │    │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                     数据存储层（Load）                        │
│        MySQL数据库：yq(舆情库) / bond(债券库) / fund(基金库)  │
└─────────────────────────────────────────────────────────────┘
```

在任务调度层面，系统根据数据更新频率和业务需求，设计了精细化的定时任务时间表：

| 时段 | 任务类型 | 典型脚本 |
|------|----------|----------|
| 05:30-06:00 | 舆情监控 | yuqing.py, jrzyq.py |
| 09:00 | 理财数据更新 | thslc.py, licaishouyi.py |
| 17:00-18:00 | 衍生品分析 | yjfxct.py |
| 20:00-23:30 | 资讯和投资者数据 | 各类资讯脚本 |
| 01:00-03:00 | 低优先级批处理 | 历史数据补充 |

### 2.2 核心方法论

本项目的方法论可以概括为"四个统一"：

**统一接口封装**：不同数据源API的差异被封装在适配器层，业务代码只需要调用统一接口。例如，无论数据来自THS还是RatingDog，最终都返回标准化的DataFrame格式。

**统一错误处理**：网络请求失败自动重试、数据库操作使用事务保护、连接池支持自动重连，确保系统的健壮性。

**统一配置管理**：数据库连接参数、API密钥等配置信息集中在config.py中管理，敏感信息通过环境变量获取，避免硬编码。

**统一增量更新**：所有数据表使用INSERT ... ON DUPLICATE KEY UPDATE语法，确保脚本可以安全地多次执行而不会产生重复数据。

### 2.3 与传统研究方式的对比

| 维度 | 传统方式 | 本项目方案 |
|------|----------|------------|
| 数据获取 | 手动从终端/网站导出 | API自动采集，定时执行 |
| 舆情处理 | 人工浏览和筛选 | AI辅助摘要和敏感词过滤 |
| 数据更新 | 逐表手动更新 | 增量更新，自动去重 |
| 异常处理 | 发现问题后手动补救 | 自动重试和错误记录 |
| 配置管理 | 分散在各脚本中 | 集中管理，环境变量隔离 |

## 3. 数据来源与处理

### 3.1 数据来源说明

本项目整合了四类主要数据源：

**THS API（同花顺）**：作为主要的金融数据来源，提供债券市场规模、期限结构、收益率曲线等市场数据。使用官方iFinDPy SDK进行对接，通过THS_DR()函数统一获取数据。该数据源更新频率高，每日可多次获取最新行情。

**RatingDog API**：提供债券发行人舆情数据，包括负面新闻、评级变动、监管处罚等信息。该API使用Bearer Token认证，需要先登录获取accessToken后再请求数据。系统配置每日4次定时获取（05:30/09:00/19:30/23:30），确保及时捕获最新舆情。

**交易所API**：从上交所等交易所官网获取公告、信息披露等数据。这类API通常需要特定的headers（Referer/User-Agent）来模拟浏览器请求，部分返回JSONP格式需要特殊解析。

**土地交易数据**：从landchina.com获取土地出让信息，用于地方政府财力分析。该数据源需要处理反爬机制，系统实现了代理IP轮换和错误重试机制。

### 3.2 数据处理要点

数据处理流程遵循"先清洗后分析"的原则，分为以下阶段：

**阶段一：原始数据获取**
- 处理API认证（Bearer Token / Cookie / headers）
- 解析返回格式（JSON / JSONP / HTML）
- 处理分页逻辑，循环获取完整数据集

**阶段二：数据清洗**
- 处理缺失值（填充默认值或标记为null）
- 类型转换（字符串日期转datetime，数值字符串转float）
- 格式标准化（统一日期格式、代码格式等）

**阶段三：AI智能处理（仅舆情数据）**
- HTML解析：get_html_text_and_images()提取纯文本和图片OCR文字
- 敏感词过滤：基于AC自动机的敏感词检测和替换
- AI摘要生成：调用大语言模型生成结构化摘要

**阶段四：数据入库**
- 使用SQLAlchemy进行ORM操作
- INSERT ... ON DUPLICATE KEY UPDATE实现增量更新
- 事务保护确保数据一致性

## 4. 核心分析框架

### 4.1 舆情监控系统

舆情监控是本项目最具技术含量的模块，实现了从数据采集到智能处理的完整闭环。

**内容处理Pipeline设计**

舆情原始数据通常是HTML格式的长文本，包含大量标签、广告等噪声。直接入库不仅占用存储空间，也不利于后续分析。因此，我们设计了四层处理Pipeline：

```
原始HTML → 第一层HTML解析 → 第二层敏感词过滤 → 第三层AI摘要 → 第四层入库存储
```

第一层HTML解析：使用get_html_text_and_images()函数，从HTML中提取纯文本内容，同时识别图片并进行OCR文字提取，确保不遗漏任何文字信息。

第二层敏感词过滤：使用AC自动机算法构建敏感词自动机，对文本进行高效扫描和替换。这一步在AI处理之前进行，既保证合规性，又减少AI处理的负担。

第三层AI摘要生成：调用timai模块的summarize_content()函数，将清洗后的内容发送给大语言模型，生成结构化摘要。摘要包含舆情涉及的公司、负面类型、发生日期、具体内容等关键信息。

第四层入库存储：将原始内容（content）和AI摘要（content_summary）分别存储在数据库中，既保留原始信息供追溯，又提供精简摘要供快速浏览。

**发行人模糊匹配算法**

舆情监控的核心价值在于将舆情与具体的债券发行人关联起来。然而，舆情中的公司名称与数据库中的发行人名称往往存在细微差异——例如"XX有限公司"与"XX有限责任公司"、"XX集团"与"XX集团有限公司"等。

为解决这个问题，我们采用基于fuzzywuzzy的模糊匹配算法：

```python
# 预处理阶段：为每个发行人建立匹配关键词集合
for issuer in issuers:
    for kw in keywords:
        if fuzz.ratio(issuer, kw.strip()) >= 95:
            issuer_to_keywords[issuer].add(kw)
```

关键设计点：
1. **预处理缓存**：在系统启动时遍历所有发行人，建立issuer_to_keywords映射字典。后续匹配直接使用缓存，将O(n*m)的复杂度降低到O(1)。
2. **阈值设定**：相似度阈值设为95%，在准确性和召回率之间取得平衡。阈值过高会漏掉变体名称，过低会产生误匹配。
3. **滚动窗口统计**：使用pandas的rolling()函数计算近180天的负面舆情累计次数，为信用风险评估提供量化指标。

### 4.2 债券指数样本计算

债券指数样本计算模块(qxqyb.py)负责从全市场债券中筛选出符合条件的样本券，用于构建各类收益率曲线。

**多维度分类体系**

样本券筛选支持以下维度组合：
- 发行人类型：城投债 / 产业债
- 发行方式：公募 / 私募
- 特殊条款：永续 / 非永续
- 信用评级：AAA / AA+ / AA / AA-等

**SQL查询优化**

样本筛选的核心是SQL查询，关键技巧包括：

1. **CROSS JOIN目标期限表**：将债券基础信息表与目标期限表进行交叉连接，为每个债券匹配最接近的目标期限点。

2. **SUBSTRING隐含评级提取**：债券隐含评级通常存储为"A-1"这样的字符串，需要使用SUBSTRING函数提取其中的主体评级部分。

3. **窗口函数排名**：使用ROW_NUMBER()窗口函数对同一发行人的多只债券进行排名，避免单一发行人在样本中占比过高。

### 4.3 理财收益跟踪

理财收益跟踪模块(licaishouyi.py)用于计算和跟踪理财产品及纯债基金的业绩表现。

**净值年化计算方法**

理财产品的净值增长率通常以期间收益率的形式呈现，需要转换为年化收益率以便比较：

```
年化收益率 = 期间收益率 × 365 / 期间天数
```

例如，某理财产品30天净值增长0.5%，则年化收益率为：0.5% × 365 / 30 = 6.08%。

**增量更新机制**

理财数据更新采用增量模式：每次执行前查询已有数据的最新日期，只补充缺失日期的数据。这种设计既减少数据传输量，又避免重复计算。

## 5. 技术/AI辅助要点

### 5.1 AI辅助舆情分析的创新应用

本项目将大语言模型应用于金融舆情处理，这是一个具有前瞻性的尝试。传统舆情处理依赖规则和关键词，对于复杂语义的理解能力有限。大语言模型的引入，使得系统能够：

1. **理解语义而非匹配关键词**：AI能够理解"公司面临流动性压力"与"公司资金链断裂"的语义差异，而非简单匹配预设的关键词。

2. **生成结构化摘要**：从长篇舆情内容中提取关键信息（涉及公司、事件类型、发生时间、影响程度），形成标准化的摘要格式。

3. **识别隐含风险**：某些舆情表面看似中性，但隐含潜在风险。AI能够识别这些隐含信息，提供风险预警。

### 5.2 多数据源整合的适配器模式

不同数据源API的认证方式、返回格式、调用限制各不相同。直接在业务代码中处理这些差异，会导致代码混乱且难以维护。适配器模式将差异封装在底层：

```python
# THS API适配器
def get_ths_data(code, indicator, start_date, end_date):
    return THS_DR(code, indicator, start_date, end_date)

# RatingDog API适配器
def get_ratingdog_data(api_path, params):
    headers = {"Authorization": f"Bearer {access_token}"}
    response = requests.get(base_url + api_path, headers=headers, params=params)
    return pd.DataFrame(response.json()['data'])
```

业务代码只需要调用统一接口，无需关心底层API的差异。当某个API变更时，只需要修改对应的适配器函数。

### 5.3 增量更新的幂等性设计

ETL脚本必须是幂等的——相同的输入产生相同的输出，多次执行不会改变最终状态。本项目通过以下设计实现幂等性：

1. **主键设计**：每个数据表设置复合主键（通常是dt + 业务键），确保同一天的同一条记录唯一。

2. **INSERT ... ON DUPLICATE KEY UPDATE**：插入时如果主键冲突则更新已有记录，而非报错或插入重复数据。

3. **数据范围查询**：每次执行前查询已有数据的日期范围，只补充缺失日期的数据。

```sql
INSERT INTO yuqing (dt, issuer, title, content, content_summary)
VALUES (%s, %s, %s, %s, %s)
ON DUPLICATE KEY UPDATE
    content = VALUES(content),
    content_summary = VALUES(content_summary);
```

这种设计确保脚本可以安全地多次执行，无论是定时任务的自动重试，还是手动补录历史数据。

## 6. 心得要点（必须有！）

### 要点一：研究基础设施的工程化思维

**背景**：

传统固收研究团队往往缺乏专职的技术支持，研究员需要自己处理数据。这导致每个人都开发了一套"自己的工具"——Excel宏、Python脚本、VBA程序等，分散且难以共享。当人员变动时，这些"部落知识"也随之流失。

**做法**：

本项目采用工程化思维，将数据采集和处理作为"研究基础设施"来建设：

1. **模块化设计**：每个脚本只做一件事，输入输出清晰定义，便于单独测试和维护。
2. **统一规范**：数据库操作统一使用SQLAlchemy，配置管理统一使用config.py，错误处理统一使用retry机制。
3. **文档化**：每个脚本头部注释说明功能、输入、输出、执行频率，便于后续维护。

**洞察**：

研究基础设施的价值在于降低边际成本。初期搭建需要投入时间，但一旦建成，后续新增数据源的成本极低——只需要编写一个适配器函数，配置定时任务即可。

好的基础设施应该是"透明的"——研究员不需要关心数据从哪里来、什么时候更新，只需要知道数据在哪里、如何使用。

**关联**：

这一心得与"增量更新的幂等性设计"和"配置管理与敏感信息保护"紧密相关——工程化思维的落地需要具体的技术实践来支撑。

### 要点二：内容处理Pipeline的分层设计

**背景**：

舆情处理涉及多个环节：HTML解析、文本清洗、敏感词过滤、AI摘要、入库存储。如果在一个大函数中完成所有步骤，代码会变得冗长且难以调试。

**做法**：

采用分层Pipeline设计，每层只做一件事：

1. **第一层（HTML解析）**：输入HTML字符串，输出纯文本
2. **第二层（敏感词过滤）**：输入纯文本，输出过滤后文本
3. **第三层（AI摘要）**：输入过滤后文本，输出结构化摘要
4. **第四层（入库存储）**：输入摘要数据，输出入库结果

每层独立测试，层与层之间通过标准化接口（字符串/DataFrame）通信。

**洞察**：

分层设计的核心原则是"单一职责"——每层只做一件事，便于理解、测试和修改。当需要更换AI模型时，只需要修改第三层，不影响其他层。

敏感词过滤放在AI处理之前是一个关键决策。这样既保证合规性（敏感内容不会发送给AI），又减少AI处理的负担（过滤后的文本更短、更干净）。

**关联**：

这一心得与"AI辅助舆情分析"密切相关——AI摘要的效果很大程度上取决于前置清洗的质量。Garbage in, garbage out。

### 要点三：幂等性是ETL脚本的第一原则

**背景**：

定时任务可能因网络问题需要重试，数据源可能更新历史数据，研究员可能需要手动补录数据。如果脚本不支持幂等执行，每次重试都会产生重复数据。

**做法**：

从设计阶段就将幂等性作为核心需求：

1. **主键设计先行**：在创建表结构时，明确哪些字段组成主键。
2. **使用UPSERT语法**：MySQL的INSERT ... ON DUPLICATE KEY UPDATE是实现幂等性的标准做法。
3. **范围查询**：每次执行前查询已有数据范围，只处理缺失部分。

**洞察**：

幂等性设计遵循"假设失败会发生"的原则。分布式系统（包括定时任务系统）必然存在失败，设计时应该考虑失败后的恢复机制。

幂等性的另一个好处是支持"回填"——当发现历史数据有问题时，可以安全地重新执行脚本覆盖错误数据，而不必担心重复。

**关联**：

这一心得与"错误处理与重试机制设计"相辅相成——重试机制的前提是脚本支持幂等执行。

### 要点四：预处理与缓存是性能优化的利器

**背景**：

发行人模糊匹配需要将每条舆情的公司名称与数千个发行人进行比对。如果每次都实时计算，O(n*m)的复杂度会导致严重的性能问题。

**做法**：

采用预处理+缓存策略：

1. **预处理阶段**：在系统启动或每日首次运行时，遍历所有发行人，为每个发行人建立匹配关键词集合。
2. **缓存结果**：将issuer_to_keywords字典保存在内存中，后续匹配直接查字典。
3. **定期刷新**：每日刷新一次缓存，确保新增发行人能够被识别。

**洞察**：

"空间换时间"是性能优化的常用策略。预处理的计算结果占用的内存空间很小（通常几MB），但带来的性能提升是数量级的。

缓存的另一个好处是"结果一致性"——同一天内，相同的输入总是产生相同的输出，便于问题排查。

**关联**：

这一心得与"定时任务的时间窗口设计"相关——预处理任务应该安排在业务查询之前完成。
