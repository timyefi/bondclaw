# BondClaw V1 发布前执行检查表

本文档把 [bondclaw_v1_closing_plan.md](bondclaw_v1_closing_plan.md) 进一步拆成可执行项。  
目标是让后续推进时可以直接按“做什么 / 完成标准 / 依赖项”逐项检查，不再停留在概念层。

## 使用方式

- P0 是发布前必须完成的内容。
- P1 是影响正式交付质量的关键内容。
- P2 是收口与保障项，尽量在发版前完成。
- 每一项都应当有明确的产物或验证结果，避免只完成“感觉上差不多”。

## P0: 发布与安装

### 1. Windows 安装包

- 产出可安装的 Windows 版本。
- 安装后能创建桌面快捷方式。
- 卸载流程可用且不会残留异常配置。
- 首次启动可正常访问 `/bondclaw` 首页。
- 完成标准：
  - 有一个可独立分发的安装包。
  - 在干净 Windows 环境中能完成安装、启动、卸载。

### 2. 自动升级最终确认

- 确认 release manifest 的字段稳定。
- 确认版本号、渠道、更新仓库、manifest URL 一致。
- 刷新后能稳定拉到远程清单。
- 升级后保留当前首页状态。
- 完成标准：
  - 应用内可看到当前版本、渠道和升级来源。
  - 升级流程不破坏品牌配置和页面状态。

### 3. 发布产物与签名

- 统一安装包、压缩包、清单文件的命名规则。
- 确认发布仓库、支持页、官网与清单配置一致。
- 如果发行渠道要求签名，则补齐签名和校验。
- 完成标准：
  - 发布产物可以被外部人员辨认和校验。
  - 产物命名、版本号、渠道说明不冲突。

## P1: Windows 原生收口

### 4. Windows 运行验证

- 在真实 Windows 环境中验证 BondClaw 首页。
- 确认执行路径都走 Windows 原生进程。
- 确认文件路径、网络访问、外链访问没有异常。
- 完成标准：
  - Windows 用户可以直接完成核心操作。
  - 关键动作能稳定运行。

### 5. 命令抽象回归

- 确认统一执行抽象覆盖全部关键动作。
- 新增命令时默认走抽象层，不直接硬编码系统命令。
- 完成标准：
  - 主要执行路径都能定位到统一抽象入口。
  - 没有新增绕过抽象层的特例。

## P1: 线索与投递

### 6. 联系方式投递链路定版

- 确认邮箱、存储、归档三路投递方案。
- 确认本地队列格式、回执状态和重试规则。
- 确认断网重试、幂等和失败回补机制。
- 完成标准：
  - 一条提交能清楚看到每个投递通道的状态。
  - 网络异常后可重试且不会重复污染数据。

### 7. 后台最小查询页

- 至少能查看提交记录、状态和重试结果。
- 不引入重型 CRM 或额外复杂服务。
- 完成标准：
  - 后台能支撑最小人工查询和排查。
  - 数据结构和桌面端本地队列一致。

## P1: 内容与体验收口

### 8. 模板中心审校

- 复核 7 个角色、20 个 starter 的命名和可读性。
- 复核输出格式和 QA 清单是否适合直接执行。
- 复核默认角色、默认模板和默认供应商是否符合真实使用习惯。
- 完成标准：
  - 模板中心可以作为正式工作入口，而不是实验目录。

### 9. 信息中心审校

- 复核 20 个案例的主题标签和角色标签。
- 复核推荐 Prompt 的映射是否准确。
- 复核来源卡、案例高亮和当前案例之间的跳转是否自然。
- 完成标准：
  - 信息中心能形成从“来源 -> 主题 -> 案例 -> Prompt”的完整链路。

### 10. 设置页最终整理

- 确认品牌、升级、在线入口、说明卡的口径统一。
- 确认设置页的视觉层级足够清楚。
- 如果需要，再做一次首屏与分区的收口。
- 完成标准：
  - 设置页能承担品牌、升级和在线入口信息展示。
  - 用户可以快速找到官网、支持页和版本来源。

## P2: 回归与文档

### 11. Smoke test

- 启动页可正常访问。
- 模板中心能切换角色和模板。
- 信息中心能按主题筛选。
- 联系方式队列可查看和重试。
- 设置页可刷新运行态。
- 完成标准：
  - 有一套最小可重复的检查流程。

### 12. 文档补齐

- 快速开始
- 常见故障
- 发布说明
- 内部维护说明
- 完成标准：
  - 新人能靠文档快速定位到使用和维护方式。

## 建议的执行节奏

1. 先做 Windows 安装包与自动升级确认。
2. 再做 Windows 原生的真实环境验证。
3. 然后定版联系方式投递链路。
4. 接着做模板中心、信息中心和设置页审校。
5. 最后补 smoke test 和文档。

## 备注

- 如果在执行中发现某项依赖没有先决条件，可以先把依赖项写到这里，再拆到更细的子任务。
- 这份检查表的目标不是“写得漂亮”，而是能真正用于推进收尾工作。
