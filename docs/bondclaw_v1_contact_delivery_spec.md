# BondClaw V1 联系方式投递链路定版

本文档用于最终确认 BondClaw 的联系方式收集与投递方式。  
它把当前已存在的 manifest、队列示例和投递策略收束成统一规范，方便发版前后都按同一套口径实现和排查。

## 目标

- 低摩擦收集联系方式。
- 多通路并行投递，避免单点失败。
- 本地队列优先，网络异常可重试。
- 让桌面端始终能看到提交状态和回执。
- 保持实现轻量化，不引入重型 CRM。

## 结论

BondClaw V1 的联系方式链路采用以下定版策略：

1. **采集时机**：在第一个有意义的核心动作时触发。
2. **本地优先**：用户提交后先落本地队列，再并行发送。
3. **三路投递**：`email -> persistent_store -> archive_link`。
4. **回执记录**：每个 sink 都记录自己的状态和回执。
5. **重试策略**：未完成 sink 按本地队列自动重试，直到成功或达到上限。
6. **source of truth**：V1 的可见状态以桌面端队列为准。

## 数据契约

联系方式提交记录至少应包含：

- `name`
- `institution`
- `role`
- `email`
- `phone`
- `card_image`
- `consent_version`
- `device_id`
- `client_version`
- `submitted_at`
- `delivery_status`
- `sink_receipts`

### `delivery_status`

推荐状态枚举：

- `queued`：已进入本地队列，等待投递。
- `partial`：部分 sink 成功，仍有 sink 待完成。
- `delivered`：全部 sink 成功。
- `failed`：已达到重试上限或全部 sink 失败。

### `sink_receipts`

每个 sink 至少记录：

- `status`
- `timestamp` 或 `record_id` / `url`
- 其他 sink 特有字段

## 三路投递定义

### 1. Email

- 用途：立即通知内部团队。
- 目标：让联系方式在最短时间内进入可读邮件流。
- 回执：记录投递是否成功、成功时间和消息标识。

### 2. Persistent Store

- 用途：稳定存储和后台查询。
- 目标：形成可检索的结构化记录。
- 回执：记录存储记录 ID 或等价主键。

### 3. Archive Link

- 用途：生成可持久访问的归档链接或安全分享链接。
- 目标：便于后续查验和归档。
- 回执：记录链接地址与生成时间。

## 重试规则

- 本地队列是重试调度的依据。
- 未完成的 sink 继续重试，不重新创建整条记录。
- 重试采用指数退避。
- 最大重试次数建议维持在 `5` 次。
- 当某个 sink 成功后，不应被下次重试重复覆盖成功状态。

## 幂等规则

- 同一条提交的幂等标识应当稳定。
- 幂等标识建议由 `device_id + submitted_at + email` 或等价组合生成。
- 重试时应复用同一提交的 sink 记录，不应生成重复的业务记录。

## 本地队列

桌面端本地队列应作为 V1 的可见状态源：

- 可以查看队列条数。
- 可以查看待重试条数。
- 可以查看每条提交的 sink 回执。
- 可以手动触发重试。

示例队列文件已经存在于：

- [financialanalysis/lead-capture/queue.example.json](../financialanalysis/lead-capture/queue.example.json)

## 轻量后台

后台不需要重型 CRM，但至少要能做到：

- 查看提交记录
- 查看 delivery_status
- 查看 sink_receipts
- 查看重试结果

如果未来要接入更正式的后台，也应尽量沿用当前字段和状态定义。

## 推荐实现顺序

1. 锁定 manifest 字段与状态枚举。
2. 保持桌面端本地队列作为主视图。
3. 定义三路 sink 的投递和回执协议。
4. 补最小查询页或轻量后台视图。
5. 再考虑更复杂的存储/消息渠道。

## 通过标准

以下条件满足时，认为联系方式链路定版：

- 三路 sink 的职责、回执和失败语义统一。
- 本地队列和后台查询看到的是同一套字段。
- 重试不会制造重复业务记录。
- 用户在网络波动下仍能看到明确状态。
- 桌面端和后台对状态含义理解一致。
