# AI辅助固收研究系列报告 第33篇：T33-信用债规模

> **研究领域**: 信用债/数据基础设施
> **研究目的**: 构建标准化的信用债规模ETL系统，解决多源数据整合与永续债久期计算难题
> **适用场景**: 日常规模跟踪/区域风险监控/行业配置分析/组合久期管理
> **更新频率**: 每日/按需

---

## 1. 研究背景与问题

### 1.1 为什么需要这项研究？

信用债市场规模分析是固定收益研究的核心基础工作。截至2024年末，我国利率债存量规模超过107万亿元，城投债发行规模占全部信用债的38%以上。随着化债政策推进和城投平台转型，市场对债券规模的结构化监测需求日益增强。

然而，传统的规模统计工作面临三个典型痛点：

**第一，数据分散且标准不统一**。信用债基础信息、金融债数据、ABS分类分别存储在不同的数据库和表中，每次做规模分析都需要重新整合数据，效率低下且容易出错。

**第二，永续债久期计算是行业难题**。永续债的期限字段以"5(+N)"、"3(+5,+5)"等特殊格式存储，表示发行人可选择续期的年限。传统的到期日计算方法完全失效，导致大量永续债的久期字段为空，影响规模统计的完整性。

**第三，评级数据时效性难以保证**。静态评级数据更新滞后，而市场隐含评级更能反映当前信用状况，但隐含评级分散在行情数据中，需要动态提取。

### 1.2 核心研究问题

本项目要回答的核心问题是：**如何构建一套标准化的信用债规模ETL系统，实现从多源异构数据到统一分析底座的自动化整合？**

具体分解为以下子问题：
- 如何建立统一的债券分类体系，覆盖城投/产业/金融/ABS四大品类？
- 如何智能解析永续债的特殊期限格式，补全久期数据？
- 如何实现多维度规模汇总，支持省/市/行业/评级/久期等15种分类方式？
- 如何设计增量更新机制，支持ETL任务的重复执行而不产生重复数据？

### 1.3 研究价值

这项研究的价值体现在三个层面：

**对投研工作的直接价值**：提供标准化的规模数据底座，投研人员无需每次从原始数据开始处理，可直接基于汇总表进行多维分析，大幅提升研究效率。

**对数据质量的保障价值**：通过统一的ETL流程，消除了人工处理带来的不一致性，确保规模数据的准确性和可追溯性。

**对研究体系的基础价值**：规模数据是信用债研究的基础设施，为后续的信用利差分析、区域风险监控、行业配置研究提供数据支撑。

## 2. 研究思路与方法

### 2.1 整体研究框架

本项目采用**分层处理、逐步完善**的ETL架构设计，将复杂的数据整合任务分解为四个相对独立的处理层：

```
┌─────────────────────────────────────────────────────────────┐
│                    数据源层（多源异构）                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐     │
│  │信用债基础表│  │金融债基础表│  │ABS分类表  │  │市场行情表  │     │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘     │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│                 第一层：分类重建层                             │
│  清空并重建bond.债券分类表，建立城投/产业/金融/ABS统一分类底座    │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│                 第二层：数据补全层                             │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │评级数据补全  │  │久期数据补全  │  │ABS类型补充  │          │
│  │（从行情获取）│  │（永续债特殊处理）│ │（跨库查询） │          │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│                 第三层：规模汇总层                             │
│  按15种分类维度聚合统计，写入bond.信用债规模表                  │
│  维度：省/市/申万一级/申万一二级/金融机构类型/ABS类型/评级/久期  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 核心方法论

#### 2.2.1 分类体系重建

传统的债券分类往往分散在多个字段中，如主体类型、发行人性质、债券条款等。本项目从原始基础信息表重建统一的分类体系，将信用债划分为四大类：

| 大类 | 判断依据 | 典型品种 |
|-----|---------|---------|
| **城投债** | 发行人属于城投平台名单 | 各省市城投平台发行的企业债、中票、短融等 |
| **产业债** | 非城投、非金融的企业发行人 | 央企、地方国企、民企发行的信用债 |
| **金融债** | 发行人属于金融机构 | 商业银行债、证券公司债、保险公司债等 |
| **ABS** | 资产支持证券标识 | 企业ABS、信贷ABS、ABN等 |

这种分类方式的优势在于：**互斥且完备**，每只债券只能归属于一个类别，且所有债券都能被覆盖。

#### 2.2.2 永续债久期智能计算

永续债久期计算是本项目最具特色的技术难点。永续债没有明确的到期日，期限字段以"5(+N)"、"3(+5,+5)"等格式存储，表示发行人有权在首次行权日赎回或选择续期。

**传统方法的局限**：直接使用到期日减去当前日期计算剩余期限，对于永续债会得到空值或错误值。

**本项目的智能解析方案**：
1. **格式识别**：使用 `LIKE '%(%)%'` 模式识别永续债的特殊期限格式
2. **数字提取**：使用嵌套的 `SUBSTRING_INDEX` 函数提取括号内首个数字（理论行权年限）
3. **结果验证**：使用 `REGEXP` 验证提取结果为纯数字
4. **久期计算**：基于起息日加上理论年限，再减去当前日期得到剩余期限

例如，期限为"5(+N)"的永续债：
- 提取理论年限：5年
- 假设起息日：2020-01-01
- 理论行权日：2025-01-01
- 若当前日期为2024-01-01，则剩余期限约为1年

#### 2.2.3 评级数据动态补全

传统做法是使用发行时的静态评级，但评级可能滞后于市场变化。本项目采用**动态评级获取**策略：

从市场行情表中获取**最新日期的隐含评级**，而非使用静态评级。隐含评级由市场定价反推得出，更能反映市场对发行人信用的即时评估。

#### 2.2.4 多维度规模汇总

项目构建了覆盖7大类15种分类方式的汇总体系：

| 分类大类 | 具体分类方式 | 应用场景 |
|---------|-------------|---------|
| 大类 | 城投/产业/金融/ABS | 总量监控 |
| 区域 | 省、市 | 区域风险分析 |
| 行业 | 申万一级、申万一二级 | 行业配置分析 |
| 机构类型 | 金融机构细分类型 | 金融债结构分析 |
| ABS类型 | ABS细分品种 | ABS市场监测 |
| 评级 | 隐含评级档次 | 信用风险监控 |
| 久期 | 久期区间 | 组合久期管理 |

### 2.3 与传统研究方式的对比

**传统方式**：
- 每次分析需要从Wind/同花顺导出原始数据
- 人工进行分类汇总，容易出错
- 永续债久期通常只能手动标注或直接忽略
- 不同分析师的分类标准可能不一致
- 历史数据难以追溯和复现

**技术辅助后**：
- 数据自动从数据库抽取，无需人工导出
- 分类逻辑固化在代码中，标准统一且可追溯
- 永续债久期实现智能解析，覆盖率大幅提升
- 增量更新机制支持每日自动运行
- 历史数据可随时重建，保证可复现性

**主要改进**（定性描述）：
- **效率提升**：原需数小时的人工整理工作，现可自动化完成
- **标准化**：消除了不同分析师分类标准不一致的问题
- **完整性**：永续债久期数据的覆盖率显著提升
- **可追溯**：所有处理逻辑固化在代码中，便于审计和修正

## 3. 数据来源与处理

### 3.1 数据来源说明

| 数据类型 | 来源 | 存储位置 | 更新频率 | 用途 |
|---------|------|---------|---------|------|
| 信用债基础信息 | 同花顺iFinD | MySQL: bond.creditinfo | 每日 | 城投/产业债分类 |
| 金融债基础信息 | 同花顺iFinD | MySQL: bond.financeinfo | 每日 | 金融债分类 |
| ABS基础信息 | 同花顺iFinD | MySQL: bond.absinfo | 每日 | ABS基础数据 |
| ABS分类数据 | 内部整理 | PostgreSQL: public.abs_type | 按需 | ABS品种分类 |
| 市场行情数据 | 同花顺iFinD | MySQL: bond.marketinfo | 每日 | 隐含评级、久期 |

### 3.2 数据处理要点

**跨数据库整合**：项目需要整合MySQL和PostgreSQL两种数据库的数据。具体做法是使用Pandas作为中间层——从PostgreSQL读取ABS分类数据后，通过DataFrame写入MySQL，实现异构数据库间的数据融合。

**久期缺失处理**：对于久期字段为空的债券，分两类处理：
- 普通债券：使用到期日减去当前日期计算
- 永续债：使用前述智能解析方案提取理论行权年限

采用**临时表批量更新模式**，先将需要更新的记录写入临时表，再通过JOIN批量更新主表，避免逐条处理带来的性能问题。

**增量更新机制**：全流程使用 `INSERT ... ON DUPLICATE KEY UPDATE` 语法，当主键冲突时更新而非报错。这种设计确保ETL任务可以重复执行而不会产生重复数据，实现了**幂等性**。

## 4. 核心分析框架

### 4.1 分析维度一：大类规模分布

**分析内容**：统计城投、产业、金融、ABS四大品类的存量规模及占比变化。

**关键发现**：
- 城投债占信用债总规模的38%以上，是信用债市场的核心组成部分
- 金融债规模近年增长较快，主要受银行资本补充债券发行推动
- ABS市场规模相对较小，但品种日益丰富

**业务含义**：
- 城投债的高占比意味着区域风险监控是信用债研究的重中之重
- 金融债的增长反映了银行补充资本的需求，需关注监管政策变化
- ABS市场的发展为投资者提供了更多元化的收益来源

### 4.2 分析维度二：区域分布与集中度

**分析内容**：按省、市维度统计城投债规模分布，识别区域集中度风险。

**关键发现**：
- 江苏省城投债规模居全国首位，区域集中度较高
- 部分西部省份城投债规模虽小，但与地方财力匹配度较低
- 地级市层面的分化程度远高于省级层面

**业务含义**：
- 省级维度监控可把握大方向，但下沉到市级才能识别真正的风险点
- 区域配置策略需在收益获取与集中度风险之间权衡

### 4.3 分析维度三：久期结构分布

**分析内容**：统计不同久期区间的规模分布，评估市场整体的久期风险暴露。

**关键发现**：
- 1年以内短期品种占比最高，反映市场对流动性的偏好
- 3-5年中等久期品种是主力配置区间
- 永续债经智能解析后，主要落入3-5年区间

**业务含义**：
- 短端集中度高意味着市场对货币政策和流动性环境高度敏感
- 组合久期管理需关注市场整体久期分布的变化趋势

## 5. 技术亮点分析

### 5.1 永续债久期智能解析

这是本项目最具特色的技术实现。金融数据中存在大量行业特有的非标准格式，永续债期限就是典型案例。

**技术实现要点**：
- 使用模式匹配识别特殊格式：`LIKE '%(%)%'`
- 嵌套字符串函数提取关键数字：`SUBSTRING_INDEX(SUBSTRING_INDEX(term, '(', -1), ')', 1)`
- 正则表达式验证提取结果：`REGEXP '^[0-9]+$'`
- 基于起息日计算理论行权日：`DATE_ADD(issue_date, INTERVAL years YEAR)`

这套方案的**可复用性**在于：类似的思路可以应用于其他非标准金融数据字段的解析，如利率重置条款、赎回条款等。

### 5.2 跨数据库数据整合

项目融合了MySQL的事务处理能力和PostgreSQL的分析能力。MySQL承载核心的债券基础信息和行情数据，PostgreSQL存储部分分析结果和分类数据。

**技术实现要点**：
- 使用Pandas的 `read_sql` 从PostgreSQL读取数据
- 通过 `to_sql` 将DataFrame写入MySQL
- 利用SQLAlchemy管理数据库连接

这种架构的**业务价值**在于：可以根据不同数据库的特性安排合适的数据存储，而不必为了统一而牺牲性能或功能。

### 5.3 ETL幂等性设计

幂等性是指：无论操作执行多少次，最终结果都是一致的。在ETL场景下，这意味着可以随时重新运行任务，而不会产生重复数据或数据不一致。

**技术实现要点**：
- 分类表使用 `TRUNCATE` 全量重建，确保每次从干净状态开始
- 汇总表使用 `INSERT ... ON DUPLICATE KEY UPDATE`，当主键冲突时更新
- 主键设计：`(DT, 分类, 子类, 分类方式)` 四字段联合主键

**业务价值**：
- 支持历史数据回溯修正：当发现历史数据有误时，只需重新运行即可
- 支持增量更新：每日运行时，已有的记录会被更新，新增的记录会被插入
- 降低运维风险：不用担心误运行导致的数据重复问题

## 6. 心得要点

### 要点一：特殊数据格式需要智能化处理方案

**背景**：在债券基础信息表中，永续债的期限字段以"5(+N)"、"3(+5,+5)"等特殊格式存储，表示发行人可选择续期的年限。传统的到期日计算方法无法处理这种格式，导致大量永续债的久期字段为空，影响规模统计的完整性。

**做法**：项目设计了一套智能解析逻辑：
1. 使用 `LIKE '%(%)%'` 识别永续债格式
2. 使用嵌套的 `SUBSTRING_INDEX` 提取括号内首个数字
3. 用 `REGEXP` 验证提取结果为纯数字
4. 基于起息日加上理论年限计算久期
5. 同时采用临时表模式批量更新缺失记录，避免逐条处理带来的性能问题

**洞察**：金融数据中存在大量行业特有的非标准格式（如债券期限、利率重置条款等），处理这类问题需要：
- **深入理解业务含义**：先搞清楚"5(+N)"在业务上代表什么，再设计技术方案
- **设计智能解析规则**：而非针对每种格式硬编码
- **建立回退机制**：当特殊格式无法解析时，使用标准方法（如到期日计算）兜底

这种"业务理解优先，技术方案跟进"的思路在金融数据处理中具有普遍参考价值。

**关联**：与"数据驱动验证"、"工程化思维"相关。

### 要点二：多源数据整合需要统一的数据底座

**背景**：项目需要整合信用债、金融债、ABS三大品类的基础信息和市场行情，这些数据分散在MySQL的多个表中，而ABS分类数据存储在PostgreSQL中。不同数据库、不同表结构给数据整合带来挑战。

**做法**：项目设计了三层整合架构：
1. **分类重建层**：清空并重建 `bond.债券分类表`，作为统一分类底座
2. **市场信息整合层**：将三大品类的市场行情统一写入 `bond.marketinfo3` 表
3. **规模汇总层**：基于整合后的数据按多维度聚合到 `bond.信用债规模表`
4. 跨数据库部分使用Pandas的 `read_sql` 和 `to_sql` 实现数据迁移

**洞察**：数据仓库建设的核心是建立统一的数据底座。这个案例展示了**"整合优先于分析"**的原则：
- 与其在每次分析时处理异构数据源，不如先投入资源建设统一的数据底座
- 后续分析都基于这个底座进行，降低复杂度和出错概率

这种前期投入虽然增加了初始成本，但能显著降低后续分析的复杂度，体现了工程化思维中"磨刀不误砍柴工"的智慧。

**关联**：与"工程化思维"、"模块化设计"相关。

### 要点三：ETL幂等性设计保障数据质量

**背景**：ETL任务通常需要定期重复执行（如每日更新），如果每次执行都追加数据，会产生大量重复记录。同时，数据源可能出现历史数据修正的情况，需要支持回溯更新。

**做法**：项目在多个环节实现了幂等性设计：
1. 债券分类表使用 `TRUNCATE` 全量重建
2. 市场信息表和规模汇总表使用 `INSERT ... ON DUPLICATE KEY UPDATE` 语法
3. 评级更新使用 `JOIN` 方式而非 `INSERT`，确保每只债券只有一条最新评级记录
4. 主键设计为 `(DT, 分类, 子类, 分类方式)` 四字段联合主键

**洞察**：幂等性是ETL系统的核心质量属性。一个设计良好的ETL流程应该满足：**无论执行多少次，最终的数据状态都是一致的**。

这不仅需要技术实现（如UPSERT语法），更需要数据模型设计（如合理的主键定义）。在金融数据场景下，幂等性设计还能支持历史数据回溯修正，这是数据质量保障的重要基础。

**关联**：与"工程化思维"、"数据驱动验证"相关。

## 7. 实际应用指南

### 7.1 如何使用这项研究
