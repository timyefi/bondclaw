# AI辅助固收研究系列报告 - 第61篇

# 指标库整理：有息债务规模计算与融资成本插值框架

---

## 摘要

在企业信用分析和债券投资研究领域，有息债务规模和融资成本是评估企业财务健康状况的核心指标。本报告详细阐述了一套基于双数据库架构（MySQL与PostgreSQL）的自动化金融指标计算框架，实现了有息债务规模的精确计算和融资成本的智能插值估算。该框架覆盖13类有息债务相关指标，采用线性回归插值算法，并通过多层验证机制确保数据的合理性和可靠性。本报告从投研视角出发，深入分析技术架构设计、核心算法原理、指标体系完整性，并总结方法论心得与改进方向。

**关键词**：有息债务、融资成本、线性插值、指标库、信用分析

## 第一章 执行摘要

### 1.1 项目概述

本项目（T61-指标库整理）旨在构建一套系统化、自动化的金融指标计算框架，解决传统方法中手动整理财务报表数据效率低下且容易出错的问题。项目的核心价值在于：

- **自动化计算**：将繁琐的手工计算流程转化为程序化执行，显著提升效率
- **标准化输出**：建立统一的指标定义和计算口径，确保数据一致性
- **智能化插值**：利用统计模型填补数据缺失，提高数据覆盖率

### 1.2 核心成果

| 成果类别 | 具体内容 |
|---------|---------|
| 指标覆盖 | 13类有息债务指标（短期8类+长期5类） |

| 核心算法 | 线性回归插值+边界约束+回退机制 |

1. **架构层面**：双数据库架构实现了OLTP（联机事务处理）与OLAP（联机分析处理）场景的有效分离，支持大规模历史数据查询
2. **方法论层面**：线性插值假设在常规市场条件下表现良好，但需关注极端市场条件下的模型有效性
3. **应用层面**：当前指标体系对传统融资工具覆盖较好，但新型融资工具和隐性债务的识别存在改进空间

## 第二章 项目背景与研究意义

### 2.1 有息债务与融资成本在信用分析中的重要性

在企业信用分析中，有息债务规模和融资成本是评估企业财务健康状况的两大核心维度。有息债务直接反映企业的债务负担水平，而融资成本则揭示企业的信用资质和市场认可度。

**有息债务规模的评估意义**：

从信用风险角度，有息债务规模决定了企业的还本付息压力。当企业有息债务过高时，即使经营现金流稳定，也可能因债务滚动困难而陷入流动性危机。近年来，国内债券市场违约案例中，相当比例的违约主体在违约前都呈现出有息债务快速攀升的特征。

**融资成本的评估意义**：

融资成本是市场对企业信用风险的定价体现。当企业融资成本持续上升时，往往预示着市场对其信用资质的担忧加剧。反之，融资成本下降则可能反映企业信用状况改善或市场流动性宽松。

### 2.2 行业痛点分析

传统方法依赖手动整理财务报表数据，存在以下问题：

| 痛点类型 | 具体表现 | 影响 |
|---------|---------|------|
| 效率问题 | 手工从财务报表中逐项提取债务数据 | 分析师大量时间消耗在数据整理上 |
| 准确性问题 | 人工计算容易出错，不同分析师口径不一 | 数据质量不稳定，横向比较困难 |
| 时效性问题 | 更新频率受限于人工处理速度 | 无法及时跟踪企业财务变化 |
| 覆盖性问题 | 难以大规模覆盖多个主体 | 样本量受限，影响统计分析可靠性 |

### 2.3 行业背景与发展趋势

2025年金融行业正经历深刻数字化转型。毕马威2025年中国银行业调查报告指出，精细化资产负债管理已成为金融机构的核心能力之一。在利率市场化深入推进、信用分化加剧的背景下，对发债主体财务状况的精准评估变得愈发重要。

从宏观层面看，非金融企业债务占GDP比重持续攀升，隐性债务化解成为政策重点。市场机构普遍采用城投债发行主体的有息债务规模来估算隐性债务规模，这进一步凸显了有息债务指标计算的重要性。

### 2.4 项目定位与目标

本项目定位为一套**可复用、可扩展的金融指标计算基础设施**，具体目标包括：

1. **标准化**：建立统一的指标定义和计算公式，确保不同分析师使用相同口径
2. **自动化**：减少人工干预，实现从数据源到计算结果的端到端自动化
3. **智能化**：利用统计方法填补数据缺失，提高指标覆盖率
4. **可追溯**：记录计算逻辑和数据来源，支持结果审计和回溯验证

## 第三章 技术架构设计

### 3.1 双数据库架构概述

项目采用MySQL与PostgreSQL双数据库架构，分别承担不同职责：

```
┌─────────────────────────────────────────────────────────────┐
│                    数据架构示意图                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────┐          ┌─────────────────────────┐     │
│   │   MySQL     │          │   PostgreSQL            │     │
│   │  (OLTP)     │          │   (TimescaleDB/OLAP)    │     │
│   ├─────────────┤          ├─────────────────────────┤     │
│   │ - 债券主数据 │  ────>   │ - 时序指标数据           │     │
│   │ - 发行主体   │  计算结果 │ - 融资成本              │     │
│   │ - 债券类型   │  写入    │ - 债券融资成本          │     │
│   │ - 基础信息   │          │ - 历史时间序列          │     │
│   └─────────────┘          └─────────────────────────┘     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 MySQL的角色与功能

MySQL作为债券基础数据存储，主要职责包括：

- **主体信息管理**：存储发行主体的基本信息，如企业名称、行业分类、注册地址等
- **债券信息管理**：存储债券的基础属性，如债券代码、发行日期、到期日期、票面利率等
- **关系映射**：建立主体与债券之间的关联关系，支持从债券维度聚合到主体维度

选择MySQL作为基础数据存储的原因在于其成熟的事务处理能力，能够保证数据的一致性和完整性。

### 3.3 PostgreSQL的角色与功能

PostgreSQL（配合TimescaleDB扩展）作为时序数据存储，主要职责包括：

- **时序数据存储**：按时间维度存储各类金融指标的历史数据
- **高效查询**：支持大规模时间范围查询和聚合计算
- **数据压缩**：利用TimescaleDB的压缩功能降低存储成本

PostgreSQL的OLAP特性使其更适合处理分析型查询，如计算时间序列统计量、生成历史趋势图表等。

### 3.4 数据流转机制

数据流转遵循"源数据读取-指标计算-结果存储"的标准流程：

1. **源数据读取**：从MySQL中提取债券和主体的基础信息
2. **外部数据接入**：接入财务数据源（如Wind、同花顺），获取各债务科目的数值
3. **指标计算**：执行有息债务汇总计算和融资成本插值
4. **结果存储**：将计算结果写入PostgreSQL的对应数据表
5. **质量校验**：对写入数据进行有效性检查，确保符合业务规则

### 3.5 配置管理方案

项目采用配置与代码分离的设计模式，关键配置独立存放在config.py中：

- **数据库连接参数**：MySQL和PostgreSQL的host、port、database、user、password
- **融资成本阈值**：如插值边界约束（2.0%-8.0%）、数据过滤阈值（3%-20%）
- **业务参数**：如有效回归系数范围（0 < coef < 1）

这种设计的优势在于：
- 配置变更不需要修改核心代码
- 敏感信息可通过环境变量管理
- 不同环境可使用独立配置文件

## 第四章 核心算法详解

### 4.1 有息债务规模计算模型

#### 4.1.1 指标分类逻辑

项目将有息债务分为短期债务和长期债务两大类别，共计13个具体指标：

**短期债务指标（8类）**：

| 序号 | 指标代码 | 指标名称 | 说明 |
|------|---------|---------|------|
| 1 | ths_st_borrow_bond | 短期借款 | 企业向银行或其他金融机构借入的期限在一年以内的借款 |
| 2 | ths_borrowing_funds_bond | 拆入资金 | 从其他金融机构拆入的资金 |
| 3 | ths_tradable_fnncl_liab_bond | 交易性金融负债 | 以公允价值计量且其变动计入当期损益的金融负债 |
| 4 | ths_derivative_fnncl_liab_bond | 衍生金融负债 | 衍生工具产生的金融负债 |
| 5 | ths_bill_payable_bond | 应付票据 | 企业购买材料、商品和接受劳务供应等开出、承兑的商业汇票 |
| 6 | ths_fnncl_assets_sold_for_repur_bond | 卖出回购金融资产款 | 通过回购协议卖出金融资产所收到的款项 |
| 7 | ths_noncurrent_liab_due_in1y_bond | 一年内到期的非流动负债 | 长期借款、应付债券等在一年内到期的部分 |
| 8 | ths_st_bond_payable_new_bond | 短期应付债券 | 期限在一年以内的应付债券 |

**长期债务指标（5类）**：

| 序号 | 指标代码 | 指标名称 | 说明 |
|------|---------|---------|------|
| 1 | ths_lt_loan_bond | 长期借款 | 期限在一年以上的银行借款 |
| 2 | ths_bond_payable_bond | 应付债券 | 企业发行的公司债券、企业债券等 |
| 3 | ths_perpetual_capital_sec_bond | 永续债 | 无固定到期日的债券，会计上可能分类为权益或负债 |
| 4 | ths_preferred_bond | 优先股 | 具有优先分配权的股权工具，但具有债券属性 |
| 5 | ths_lease_libilities_bond | 租赁负债 | 新租赁准则下确认的租赁负债 |

#### 4.1.2 计算公式

有息债务规模的计算公式如下：

$$
\text{有息债务} = \sum_{i=1}^{8} \text{短期债务}_i + \sum_{j=1}^{5} \text{长期债务}_j
$$

在实际计算中，需要注意以下细节：
- 空值处理：对于缺失的科目数据，按0处理（假设该科目无余额）
- 合并口径：使用合并报表数据，反映集团整体债务情况
- 期间匹配：确保各科目数据来自同一报告期

### 4.2 融资成本线性插值算法

#### 4.2.1 算法背景

融资成本是评估企业信用状况的重要指标，但并非所有主体都有公开披露的融资成本数据。本项目利用债券融资成本（可公开计算）与综合融资成本之间的相关关系，通过线性回归进行插值估算。

#### 4.2.2 核心假设

算法基于以下核心假设：
- 债券融资成本与综合融资成本之间存在稳定的线性关系
- 同一主体在不同时期的成本关系具有延续性
- 市场条件相对稳定，未发生结构性变化

#### 4.2.3 算法步骤

算法的具体执行步骤如下：

**步骤1：数据准备**
- 对同一主体的历史数据按债券融资成本排序
- 提取已知融资成本的数据点作为训练样本

**步骤2：模型拟合**
- 使用sklearn的LinearRegression拟合模型
- 模型形式：$\text{综合融资成本} = \alpha + \beta \times \text{债券融资成本}$

**步骤3：系数验证**
- 检查回归系数$\beta$是否在(0, 1)区间内
- 系数大于0：债券融资成本上升时，综合融资成本同向变化
- 系数小于1：债券融资成本的波动幅度大于综合融资成本（合理，因为综合融资成本包含其他相对稳定的融资来源）

**步骤4：插值预测**
- 对缺失融资成本的数据点，使用拟合模型进行预测
- 应用边界约束：将预测结果限制在2.0%-8.0%区间

**步骤5：回退机制**
- 当回归系数不满足(0, 1)条件时，使用简化公式：
$$
\text{综合融资成本} = \frac{\text{债券融资成本} - \text{基准成本}}{3} + \text{基准融资成本}
$$

#### 4.2.4 边界约束的经济学解释

将融资成本限制在2.0%-8.0%区间的设置基于以下考虑：

- **下限2.0%**：低于此水平的融资成本可能不符合市场实际，因为即使是最优质的央企也难以获得如此低的综合融资成本
- **上限8.0%**：超过此水平可能反映数据异常或主体已陷入严重财务困境，需要人工复核

### 4.3 数据质量校验机制

#### 4.3.1 多层过滤机制

项目建立了多层过滤机制确保数据质量：

| 层级 | 校验规则 | 目的 |
|------|---------|------|
| 第一层 | 债券融资成本非空检查 | 排除无法计算的基础数据 |
| 第二层 | 融资成本范围校验（3%-20%） | 过滤明显异常的原始数据 |
| 第三层 | 债券融资成本上限过滤（<=20%） | 剔除极端值对回归的影响 |
| 第四层 | 插值结果边界约束（2%-8%） | 确保输出结果的合理性 |

#### 4.3.2 校验逻辑的设计原则

数据校验逻辑的设计遵循以下原则：

1. **经济逻辑优先**：校验规则应基于经济学常识，而非纯统计方法
2. **分层处理**：不同层级的问题采用不同的处理方式（剔除、修正、标记）
3. **可追溯性**：记录被过滤的数据及其原因，支持后续审计
4. **适度宽松**：避免过度过滤导致样本量大幅减少

## 第五章 指标体系说明

### 5.1 短期债务指标详解

#### 5.1.1 传统短期融资工具

**短期借款**：这是企业最常见的短期融资方式，包括信用借款、保证借款、抵押借款等。短期借款的期限通常在一年以内，利率根据企业信用资质和市场资金面情况确定。在分析时，需要关注短期借款的期限分布和续贷风险。

**应付票据**：包括银行承兑汇票和商业承兑汇票。银行承兑汇票由银行担保，信用风险较低；商业承兑汇票依赖企业自身信用，风险相对较高。应付票据在本质上是一种延迟支付手段，但也构成企业的短期债务负担。

#### 5.1.2 金融市场融资工具

**拆入资金**：主要指从银行间市场拆入的资金，通常是金融机构或大型企业的融资渠道。拆入资金的期限较短（通常隔夜至数月），利率与Shibor等市场基准利率挂钩。

**卖出回购金融资产款**：企业通过回购协议卖出债券等金融资产获得资金。这是一种抵押融资方式，融资成本通常低于信用借款。需要关注的是，如果企业频繁使用卖出回购融资，可能反映其常规融资渠道受限。

#### 5.1.3 现代金融工具

**交易性金融负债**和**衍生金融负债**：这两类负债与企业的金融市场交易活动相关。交易性金融负债可能来自做空操作或金融资产的公允价值变动；衍生金融负债则来自利率互换、外汇远期等衍生工具的估值。

### 5.2 长期债务指标详解

#### 5.2.1 传统长期融资工具

**长期借款**：企业向银行或其他金融机构借入的期限在一年以上的借款。长期借款通常用于固定资产投资或项目融资，还款来源主要是项目现金流或企业经营现金流。分析时需要关注借款的期限结构、利率类型（固定/浮动）和担保方式。

**应付债券**：企业发行的公司债券、企业债券、中期票据等。债券融资具有金额大、期限长、利率相对稳定的优点。需要关注债券的到期分布、提前赎回条款和交叉违约条款。

#### 5.2.2 混合资本工具

**永续债**：无固定到期日的债券，发行人可无限期推迟支付本金。永续债在会计上可能分类为权益工具或金融负债，取决于具体条款设计。对于投资者而言，永续债具有"类股权"属性，其清偿顺序通常排在普通债务之后。

**优先股**：具有优先分配股利和剩余财产权利的股权工具。优先股股利通常是固定的，具有债券的固定收益特征。在计算有息债务时，是否将优先股纳入需要根据具体条款判断。

#### 5.2.3 租赁相关

**租赁负债**：根据新租赁准则（CAS 21），承租人需要确认租赁负债，反映未来租赁付款额的现值。租赁负债的增加会导致企业资产负债率上升，但这只是会计处理变化，实际债务负担可能并未增加。

### 5.3 指标覆盖的完整性分析

#### 5.3.1 已覆盖的融资工具

当前指标体系对传统融资工具覆盖较好，包括：
- 银行信贷类：短期借款、长期借款
- 债券类：应付债券、短期应付债券
- 金融市场类：拆入资金、卖出回购
- 混合工具类：永续债、优先股
- 租赁类：租赁负债

#### 5.3.2 潜在的覆盖缺口

需要注意的是，以下类型的融资可能未完全覆盖：

| 融资类型 | 说明 | 识别难度 |
|---------|------|---------|
| 供应链金融 | 核心企业通过商业信用为上下游提供融资 | 中等 |
| 明股实债 | 名义股权投资，实际具有债务属性 | 高 |
| 表外融资 | 未在资产负债表内反映的融资承诺 | 高 |
| 关联方借款 | 集团内部或关联方的借款 | 中等 |

## 第六章 可视化分析

### 6.1 融资成本与债券融资成本关系分析

通过可视化分析融资成本与债券融资成本之间的关系，可以验证线性插值假设的合理性。

**散点图分析**：

在散点图中，横轴表示债券融资成本，纵轴表示综合融资成本。理想情况下，数据点应呈现明显的正相关关系，且回归线的斜率在0到1之间。

**回归诊断**：

通过观察回归残差的分布，可以评估线性假设的适用性。如果残差呈现随机分布，说明线性模型适用；如果残差呈现明显的模式（如U型分布），则可能需要考虑非线性方法。

### 6.2 融资成本分布特征

**整体分布**：

融资成本的分布通常呈现右偏特征，大部分主体的融资成本集中在较低水平，少数主体具有极高的融资成本。这与信用评级的分布特征一致：投资级主体数量多于投机级主体。

**行业差异**：

不同行业的融资成本存在显著差异。公用事业、交通运输等稳定性行业的融资成本通常较低；房地产、民营企业等风险较高行业的融资成本相对较高。

### 6.3 关键统计量

| 统计量 | 含义 | 应用场景 |
|-------|------|---------|
| 均值 | 行业平均融资成本 | 整体水平评估 |
| 中位数 | 行业中位融资成本 | 排除极端值影响后的代表值 |
| 标准差 | 融资成本离散程度 | 行业内部分化程度 |
| 分位数 | 不同百分位水平 | 识别异常高/低成本主体 |

## 第七章 方法论心得

### 7.1 金融指标插值应遵循"先验证后应用"原则

本项目在融资成本插值中引入了回归系数验证机制（0 < coef < 1），这是一种重要的方法论实践。金融数据往往存在噪声和异常值，直接应用统计模型可能导致不合理的外推结果。通过设定系数有效区间，可以过滤掉违背经济逻辑的模型结果。

**验证机制的经济学含义**：

- 系数大于0：确保债券融资成本与综合融资成本同向变化
- 系数小于1：确保综合融资成本的波动幅度小于债券融资成本（因为综合融资包含银行贷款等相对稳定的融资来源）

**扩展应用建议**：

在其他金融指标插值场景中，也应建立类似的经济逻辑校验机制：

1. **收益率插值**：应保持期限结构的单调性（正常情况下长期收益率高于短期）
2. **信用利差插值**：应保持评级层级关系（评级越低，利差越大）
3. **流动性指标插值**：应考虑市场深度特征（大盘股流动性通常好于小盘股）

### 7.2 指标分类体系应与监管披露口径对齐

项目将有息债务分为短期和长期两大类共13个指标，这种分类方式与Wind、同花顺等数据源的披露口径基本一致。这种对齐确保了数据的可追溯性和可比较性，是构建指标库的基础性工作。

**对齐的重要性**：

- **数据可追溯**：当出现异常数据时，可以追溯到原始来源进行核实
- **横向可比较**：不同分析师使用相同口径，确保研究结果的可比性
- **监管合规**：符合监管机构对信息披露的要求

**扩展建议**：

在扩展指标库时，建议：

1. 参考CSMAR、RESSET等主流数据库的指标定义
2. 建立指标与会计准则（CAS/IFRS）的映射关系
3. 记录指标计算公式和取数逻辑，便于后续审计和回溯

### 7.3 配置与代码分离是可维护性的关键

项目将数据库连接参数、融资成本阈值等关键配置独立存放在config.py中，通过环境变量实现敏感信息保护。这种设计使得配置变更不需要修改核心代码，提高了系统的可维护性和安全性。

**设计原则**：

- **单一职责**：配置文件只负责参数定义，不包含业务逻辑
- **环境隔离**：不同环境（开发/测试/生产）使用独立配置
- **敏感信息保护**：密码等敏感信息通过环境变量管理

**进一步优化建议**：

1. 使用.env文件管理环境变量，避免敏感信息进入代码仓库
2. 对不同环境使用独立配置文件（如config.dev.py、config.prod.py）
3. 敏感信息考虑使用密钥管理服务（如AWS Secrets Manager、HashiCorp Vault）

### 7.4 回退机制设计的重要性

在融资成本插值算法中，当回归系数不满足条件时，使用简化公式作为回退方案。这种设计确保了算法在各种情况下都能产出结果，而不是简单地报错退出。

**回退机制的设计原则**：

- **保底可用**：即使主算法失效，也要有可用的备选方案
- **结果合理**：回退方案产出的结果应在合理范围内
- **标记可追溯**：标记使用回退方案的数据，便于后续分析

## 第八章 局限性与改进方向

### 8.1 当前局限性分析

#### 8.1.1 数据源依赖

项目依赖特定的数据库表结构（如"融资成本1"表），迁移到其他环境需要重建数据表。这增加了系统部署和维护的复杂度。

**影响分析**：
- 新用户接入成本高
- 数据表结构变更可能导致程序失效
- 难以在多环境间复用

#### 8.1.2 插值方法局限

线性插值假设成本关系为线性，在极端市场条件下可能失效。

**具体表现**：
- 信用危机期间，债券融资成本可能飙升，但银行贷款等渠道的融资成本上升幅度有限
- 此时线性外推可能低估综合融资成本

#### 8.1.3 异常处理不足

代码中异常处理较为简单，主要打印错误信息，缺乏重试机制和详细日志。

**潜在风险**：
- 临时性网络故障可能导致任务失败
- 错误信息不完整，难以定位问题根源

#### 8.1.4 单元测试缺失

项目为Notebook形式，缺乏自动化测试用例。

**影响分析**：
- 代码修改后难以验证功能正确性
- 回归问题可能在生产环境才被发现

#### 8.1.5 配置安全性

config.py中包含硬编码的数据库密码，存在安全风险。

**潜在风险**：
- 代码泄露可能导致数据库被非法访问
- 不符合安全审计要求

### 8.2 改进方向建议

#### 8.2.1 短期改进（1-2周）

### 9.2 使用场景详解

#### 场景一：企业信用分析

**分析目标**：通过计算有息债务规模和融资成本，评估企业债务负担和再融资能力。

**关键指标**：
- 有息债务/EBITDA：衡量债务相对于盈利能力的水平
- 融资成本变化趋势：反映市场对企业信用状况的认知变化
- 债务期限结构：评估短期偿债压力

**分析流程**：
1. 提取目标主体的财务数据
2. 计算有息债务规模及其构成
3. 分析融资成本的历史变化
4. 与同行业可比公司进行横向比较
5. 形成信用评估结论

#### 场景二：债券投资决策

**分析目标**：对比发行主体融资成本与市场收益率，识别定价偏差。

**关键指标**：
- 融资成本vs债券发行利率：判断债券定价是否合理
- 同行业融资成本比较：识别行业内的相对价值

**分析流程**：
1. 获取目标债券的基本信息
2. 计算发行主体的融资成本
3. 比较融资成本与债券收益率
4. 分析利差变化的驱动因素
5. 形成投资建议

#### 场景三：行业风险监测

**分析目标**：汇总行业融资成本数据，监测系统性风险变化。

**关键指标**：
- 行业平均融资成本：反映行业整体信用状况
- 融资成本分化度：反映行业内信用分层程度

**分析流程**：
1. 确定目标行业范围
2. 汇总行业内所有主体的融资成本
3. 计算行业统计量（均值、中位数、分位数）
4. 跟踪时间序列变化
5. 识别风险信号

### 9.3 扩展建议

基于当前框架，可以考虑以下扩展方向：

1. **增加WACC计算模块**：加权平均资本成本是企业价值评估的重要参数
2. **引入行业分类进行分层分析**：不同行业的融资成本特征差异显著
3. **构建融资成本预警阈值体系**：设置预警线，自动触发风险提示
4. **对接Wind/Choice等外部数据源进行数据校验**：提高数据质量
