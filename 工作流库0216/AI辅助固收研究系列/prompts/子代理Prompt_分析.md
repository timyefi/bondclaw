## 执行步骤

### Step 1: 读取探索结果

使用Read工具读取探索阶段输出的JSON文件：
```
G:\我的云端硬盘\workspace\projects\工作库\AI辅助固收研究系列\temp\{TXX}_探索结果.json
```

### Step 2: 深度代码分析

根据探索结果，**选择性读取关键代码文件**：
- 重点读取核心算法和处理逻辑
- 理解数据处理流程
- 不要读取所有文件，只关注重点

### Step 3: 网络搜索补充

使用WebSearch工具进行补充搜索：
- 基于探索结果中的搜索关键词
- 搜索行业最佳实践和方法论
- 搜索相关的监管要求（如适用）

### Step 4: ⭐ 心得要点提炼（关键！）

从项目中提炼1-3个方法论层面的心得。回答以下问题：

```
┌─────────────────────────────────────────────────────────────┐
│                   心得要点提炼表                              │
├─────────────────────────────────────────────────────────────┤
│ Q1: 这个项目体现了什么方法论层面的洞察？                        │
│     （不是技术细节，而是"为什么这么做的道理"）                  │
├─────────────────────────────────────────────────────────────┤
│ Q2: 遇到的困难，用什么思路解决的？这个思路能否复用？             │
├─────────────────────────────────────────────────────────────┤
│ Q3: 设计决策中有什么值得借鉴的经验？                           │
├─────────────────────────────────────────────────────────────┤
│ Q4: 与已知心得（工程化、重复冗余、多模型评估等）的关联？          │
└─────────────────────────────────────────────────────────────┘
```

### Step 5: 设计报告结构

基于分析结果，建议报告的重点章节：
- 哪些章节应该重点写？
- 哪些章节可以简写或跳过？
- 字数如何分配？

## 核心心得要点知识库

以下是已知的核心心得要点，分析时应关联和扩展：

| 心得要点 | 核心内容 |
|---------|---------|
| **工程化思维** | AI和技术与人协同工作的本质是工程。工程化能够帮助提高效率、规范工作、提升质量、确保可复现性 |
| **重复冗余设计** | AI本质是概率论，出错和考虑不全面是必然的。通过多次重复，能够极大程度减少错误 |
| **多模型评估** | 设计阶段使用不同的大模型来反复评估方案，不同模型的视角和盲点不同，交叉验证能发现更多问题 |
| **模块化设计** | 复杂系统要拆分为独立模块，每个模块职责单一 |
| **数据驱动验证** | 任何结论都需要数据支撑，通过数据验证假设 |
| **迭代优化** | 先有基础版本，再逐步优化 |
| **可追溯性** | 代码和分析过程要能追溯 |

## 注意事项

- 这是分析阶段，**不要写报告**
- 深度分析核心代码，但不要读取所有文件
- 心得要点是报告的核心价值，必须认真提炼
- 网络搜索结果用于补充背景，不是项目的具体数据
- 如果某些信息无法获取，填写"未找到"而非编造

---

## ⚠️ 禁止清单

分析时避免以下错误：

❌ 编造项目没有的功能
❌ 夸大项目的作用
❌ 使用通用的、万金油式的描述
❌ 没有依据的定量描述（如"效率提升5倍"）
