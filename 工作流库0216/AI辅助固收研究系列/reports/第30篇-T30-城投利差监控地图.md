## 1. 研究背景与问题

### 1.1 为什么需要这项研究？

城投债信用利差的区域分化是近年来债券市场的显著特征。根据联合资信和用益信托网的研究报告，2025年城投债市场呈现明显的区域分化特征，资金向高信用、强区域集中，江苏、山东、浙江三省发行规模占全国40%。同时，利差分化加剧，长久期品种信用分化明显。

在此背景下，区域城投风险"地图化"已成为行业认可的分析方式。标普信评发布的"城投压力地图"报告便是典型代表。市场上虽有DM数据终端等商业化产品提供城投利差监控功能，但多为付费终端，且数据格式和展示方式难以定制。

对于投研团队而言，核心需求是：**如何快速、直观地识别哪些区域利差异常扩大？哪些区域存在投资机会或风险？** 传统的表格数据难以满足"一眼看清全国"的需求。

### 1.2 核心研究问题

- **问题一**：如何将分散在各省份、各期限的城投利差数据，转化为直观的地图可视化？
- **问题二**：如何实现从全国到省份的"下钻"分析，既能看全局，又能聚焦细节？
- **问题三**：如何确保热力图的颜色编码准确传达信息，而非产生视觉误导？

### 1.3 研究价值

- **快速识别异常区域**：通过热力图颜色变化，一眼发现利差异常扩大或收窄的区域
- **支持投资决策**：辅助投研人员快速定位潜在投资机会或需要规避的风险区域
- **提升汇报效率**：交互式地图比传统表格更直观，便于向非技术背景的客户或管理层展示

## 2. 研究思路与方法

### 2.1 整体研究框架

本项目的核心是构建"数据转换-格式标准化-可视化映射"的三阶段处理流程：

```
原始曲线数据(hzcurve_credit)
    |
    v
SQL窗口函数计算(7天/30天滚动变化)
    |
    v
加权平均聚合(省/市两级)
    |
    v
名称标准化处理
    |
    v
Highcharts地图可视化
    |
    v
交互式下钻(全国->省份)
```

### 2.2 核心方法论

**方法论一：加权平均利差计算**

区域综合利差的计算并非简单平均，而是以债券余额(balance)为权重进行加权平均。这确保了大额债券对区域利差的影响权重更高，更能反映区域实际的融资成本。

公式表达：
```
区域加权平均利差 = (单券利差1 x 债券余额1 + 单券利差2 x 债券余额2 + ...) / (债券余额1 + 债券余额2 + ...)
```

**方法论二：窗口函数实现时序聚合**

使用SQL窗口函数（FIRST_VALUE/LAST_VALUE OVER PARTITION BY）计算7天和30天的滚动收益率变化。这种方式直接在数据库层面完成计算，避免了多次查询和数据传输的开销。

**方法论三：省市两级下钻机制**

前端通过动态加载GeoJSON实现地图层级切换，同时动态更新颜色轴范围。用户可以从全国地图点击进入某省份，查看该省份内各市的详细利差分布。

### 2.3 与传统研究方式的对比

**传统方式**：
- 投研人员从多个数据源（Wind、DM、中债等）导出Excel表格
- 手动计算各区域利差变化
- 通过Excel条件格式或人工标注识别异常
- 制作PPT汇报时需要手动更新图表

**技术辅助后**：
- 数据自动从数据库抽取并聚合
- 利差变化通过SQL窗口函数自动计算
- 热力图自动生成，异常区域直观可见
- 支持交互式下钻，点击即可查看详细数据

**主要改进**：
- 从"静态表格"到"动态地图"，信息密度大幅提升
- 从"人工计算"到"自动聚合"，减少人为错误
- 从"全局模糊"到"可下钻细看"，兼顾宏观与微观

## 5. 技术辅助要点

### 5.1 SQL窗口函数的"计算下推"策略

本项目使用PostgreSQL窗口函数直接计算滚动窗口的起始值和结束值：

```sql
FIRST_VALUE(stdyield) OVER (
    PARTITION BY province
    ORDER BY dt
    RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW
) as yield_7d_ago
```

**为什么这样做？**
- 传统方案需要分别查询多个时间点的数据，然后在应用层计算差值
- 窗口函数将计算"下推"到数据库层，一次查询即可获取所有指标
- 利用数据库查询优化器提升性能，减少数据传输量

**关键细节**：使用`RANGE`而非`ROWS`
- 金融时间序列存在非交易日（周末、节假日）
- `ROWS BETWEEN 7 PRECEDING`会严格取前7条记录，可能跨越超过7个自然日
- `RANGE BETWEEN INTERVAL '7 days' PRECEDING`会正确计算7个自然日前的数据

### 5.2 热力图颜色轴的动态更新

从全国地图下钻到省份地图时，如果颜色轴范围保持不变，会导致省内各市利差差异不明显（因为省内利差波动范围通常小于全国范围）。

**解决方案**：
- 后端预先计算各省份的max/min值（province_stats）
- 下钻时根据当前省份的统计值更新colorAxis
- 返回上级时恢复全国范围的colorAxis

### 5.3 前后端数据格式的"契约设计"

原始数据中省份名称格式不统一，导致地图渲染失败。与其在前端做复杂的数据适配，不如在后端输出时就严格遵循契约格式。

**标准化转换逻辑**：
1. 分析目标格式（MySQL临时表的数据结构）
2. 建立标准化的名称转换规则
3. 将转换逻辑封装为可复用函数

## 6. 心得要点

### 要点一：前端可视化组件与后端数据格式的"契约设计"

**背景**：项目中需要将后端计算的利差数据传递给Highcharts地图组件，两者的数据格式必须完全匹配。原始数据中省份名称格式不统一（有的带"省"，有的带"自治区"），导致地图渲染失败。

**做法**：通过分析目标格式（MySQL临时表temp_5843/temp_5857的数据结构），建立了标准化的名称转换规则：
1. 统一截取前2个字符作为简称
2. 对内蒙古、黑龙江等3字简称做特殊处理
3. 将这种转换逻辑封装为可复用的函数

**洞察**：在前后端分离的数据可视化项目中，数据格式是前后端的"契约"。与其在前端做复杂的数据适配，不如在后端输出时就严格遵循契约格式。这种"契约优先"的设计思维可以大幅减少前后端联调的沟通成本和错误率。

具体而言：
- **为什么有效**：后端统一处理一次，前端无需关心数据清洗逻辑，职责边界清晰
- **背后原理**：模块化设计原则——每个模块只做一件事，数据转换归后端，展示归前端
- **复用场景**：任何涉及前后端数据交互的项目，都应先定义清楚"契约"，再开发实现

**关联**：与"模块化设计"和"数据驱动验证"方法论一致——先定义数据格式（契约），再验证输出是否符合契约。

### 要点二：热力图下钻时的"视觉一致性"问题

**背景**：从全国地图下钻到省份地图时，如果颜色轴范围保持不变，会导致省内各市利差差异不明显（因为省内利差波动范围通常小于全国范围）。

**做法**：在代码中实现了动态颜色轴更新：
1. 后端预先计算各省份的max/min值（province_stats）
2. 下钻时根据当前省份的统计值更新colorAxis
3. 返回上级时恢复全国范围的colorAxis

**洞察**：热力图的有效性依赖于颜色编码的信息密度。如果颜色范围不能动态适应数据特征，就会导致信息丢失。

- **为什么有效**：用户在不同层级看到的是"相对差异"而非"绝对值"，这更符合人类视觉感知的规律
- **背后原理**：格式塔心理学——人们倾向于感知整体中的相对关系，而非孤立判断绝对值
- **复用场景**：任何多层级下钻的热力图/色阶图都应考虑动态调整颜色范围

**关联**：与"迭代优化"方法论相关——先有基础版本，再根据用户反馈（"为什么下钻后看不出差异"）优化颜色轴逻辑。

### 要点三：窗口函数在金融时序分析中的"计算下推"优势

**背景**：需要计算各区域7天和30天的滚动收益率变化。传统方案是分别查询多个时间点的数据，然后在Pandas中计算差值。

**做法**：使用PostgreSQL的窗口函数（FIRST_VALUE/LAST_VALUE OVER PARTITION BY ORDER BY dt RANGE BETWEEN INTERVAL）在SQL层面直接计算滚动窗口的起始值和结束值，一次查询即可获取所有需要的指标。

**洞察**：将复杂计算"下推"到数据库层是处理大规模金融数据的有效策略。

- **为什么有效**：SQL窗口函数可以利用数据库的查询优化器、索引、并行执行等能力，比在应用层逐行处理更高效
- **背后原理**：计算下推（Pushdown）——将计算推到数据所在的位置，而非将数据拉到计算所在的位置
- **复用场景**：任何涉及时序窗口计算的场景（移动平均、环比同比、滚动统计）都应优先考虑SQL窗口函数

**关联**：与"工程化思维"方法论一致——选择合适的工具在合适的层级完成任务，而不是盲目地在应用层处理所有逻辑。
