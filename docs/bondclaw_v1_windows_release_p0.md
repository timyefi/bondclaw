# BondClaw V1 Windows 发布 P0 步骤

本文档专注于 P0 里的两件事：

- Windows 安装包
- 自动升级最终确认

它的目标是把“要做什么”拆成“按步骤怎么做、做完怎么看算过”。

## 使用前提

- 已完成基础前端收口。
- 已有可运行的 BondClaw 首页。
- 已有 release manifest、品牌配置和更新配置。
- 已确认 Windows 端最终运行路径走原生进程。

## 1. Windows 安装包

### 1.1 打包前检查

- 确认当前主分支可正常启动。
- 确认 `docs/bondclaw_v1_smoke_test.md` 已通过。
- 确认 `docs/bondclaw_v1_contact_delivery_spec.md` 已定版。
- 确认品牌配置、升级配置和首页都已统一。

### 1.2 构建安装产物

- 生成 Windows 安装包。
- 生成与版本对应的发布清单。
- 生成必要的校验文件或签名材料。

### 1.3 安装验证

- 在干净 Windows 机器上安装。
- 确认安装过程无异常弹窗。
- 确认安装后可创建桌面快捷方式。
- 确认首次启动可直接进入 `/bondclaw`。

### 1.4 卸载验证

- 执行卸载。
- 确认卸载流程可完成。
- 确认卸载后不会残留明显异常配置或损坏快捷方式。

### 1.5 安装包通过标准

- 干净 Windows 环境可安装。
- 安装后可启动。
- 卸载后可恢复干净状态。
- 安装包命名、版本号、渠道与清单一致。

## 2. 自动升级最终确认

### 2.1 清单确认

- 确认 release manifest 字段完整。
- 确认版本号、渠道、更新仓库和 manifest URL 一致。
- 确认默认升级入口仍指向当前发布仓库。

### 2.2 刷新确认

- 在应用内刷新运行态。
- 确认可稳定拿到远程清单。
- 确认刷新后品牌、设置和页面状态未异常变化。

### 2.3 升级确认

- 从旧版本升级到新版本。
- 确认升级前后版本号正确变化。
- 确认升级后首页、模板中心、信息中心、联系方式、设置页均可访问。
- 确认升级后本地队列和首页状态保留。

### 2.4 回退与恢复

- 如升级失败，确认可回到原版本或恢复上一个可用状态。
- 确认升级异常不会破坏当前首页数据。

### 2.5 升级通过标准

- manifest、版本号、渠道和更新仓库一致。
- 刷新清单稳定。
- 升级前后首页状态保留。
- 升级不会破坏品牌配置和内容页状态。

## 3. 推荐验收顺序

1. 先确认打包前检查通过。
2. 再生成 Windows 安装产物。
3. 然后做干净环境安装和卸载。
4. 再做 release manifest 和刷新验证。
5. 最后做升级与回退验证。

## 4. 记录建议

每次执行建议记录：

- 安装包版本
- Windows 版本
- 原生执行状态
- 更新仓库
- manifest URL
- 首次启动结果
- 升级前后版本号
- 失败截图或日志

## 5. 结论

只要这份 P0 步骤跑通，BondClaw 就具备正式发布前最关键的 Windows 交付能力。
