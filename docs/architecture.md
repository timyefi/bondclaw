# 架构说明

这个项目的核心不是单脚本，而是一条分层工作流。每一层只解决一个问题，避免“一个脚本包办全部”。

V1 的桌面运行约束是：

- 客户端是 Windows 原生桌面应用。
- 客户安装后不依赖 Linux 发行版或外部终端环境。
- 命令模板默认按 Windows 原生执行口径编写。
- UI 层和技能层要分离，桌面壳只负责编排和展示。

## 分层

1. 采集层：从公开来源拿到原始材料。
2. 解析层：把 PDF、Word、PPT 或图片转成可读中间文本。
3. 标准化层：识别主体、期间、附注范围和章节结构。
4. 分析层：按附注优先原则分析主体的财务结构、债务压力、现金覆盖、受限资金、担保、或有事项和条款风险，并生成工作底稿和报告。
5. 导出层：把稳定结果放进对外交付格式。
6. 运行治理层：记录 manifest、状态、重跑和失败原因。
7. 知识治理层：管理正式知识库与回滚。

## 为什么要这样拆

- 采集和解析经常失败，但失败原因不同。
- 标准化和分析需要不同的人读不同的中间产物。
- 对外交付内容不能和内部知识演进混在一起。
- 后续团队协作时，边界越清楚，越不容易互相污染。

## Windows 原生原则

- 客户可见动作必须能直接在 Windows 桌面环境完成。
- 任何内部兼容路径都不能成为安装前提。
- 任何运行时提示都应优先使用 Windows 语言描述，而不是 shell 语言。

## 核心技能的角色

- `financialanalysis/` 负责承接核心技能与参考实现。
- 项目级文档负责解释边界、契约和协作方式。
- `financialanalysis/research-writing/` 负责统一的研报写作风格技能。

## 文档边界

- 项目内只保留开发与协作必须的文档。
- 长篇业务说明不要放进仓库主干文档里。
